1 类的加载、连接和初始化

​ 系统可能在第一次使用某个类时加载该类，也可能采用预加载机制来加载某个类。

1.1 JVM和类

​ 当运行某个Java程序时会启动一个Java虚拟机进程，不管该Java进程有多么复杂，该程序启动了多少个线程，它们都处于Java虚拟机进程里。也就是说，同一个JVM的所有线程、所有变量都处于同一个进程里，它们都使用该JVM进程的内存区。 当系统出现以下几种情况时，JVM进程将被终止：

  • 程序运行到最后正常结束。
  • 程序执行过程中遇到了未捕获的异常或错误而结束。
  • 程序运行到使用了System.exit()或Runtime.getRuntime().exit()方法的代码处结束程序。
  • 程序所在的平台强制结束了JVM进程。

​ 从上面的介绍可以看出，当Java程序运行结束时，JVM进程结束，该程序在内存中的状态将会丢失。下面以类的静态变量来说明这个问题。

  1. 先定义一个包含静态变量的类。
  1. 定义一个类创建A类的实例，并访问A对象的静态变量a。
  1. 下面程序也创建A对象，并访问其变量a的值

​ 在ATest1.java程序中创建了A类的实例，并让该实例的静态变量a的值自增1，程序输出的结果为8，这是显而易见的。关键是运行第2个程序ATest2时，程序再次创建了A对象，并输出A对象静态变量a的值，此时a的值是多少呢？

​ 相信很多读者会毫不犹豫的认为是8，因为A类里面的a变量是静态变量，同一个类的所有实例的静态变量共享同一块内存区，因为第一次运行时改变了第一个A实例的a变量，所以第二次运行时第二个A实例的a变量也将受到影响。

​ 实际上结果依然是7，而不是8。这是因为ATest1和Atest2是两次运行JVM进程，第一次运行JVM结束后，它对A类所做的修改将全部丢失——第2次运行JVM时将再次重新初始化A类。两次运行的JVM程序处于两个不同的JVM进程中，两个JVM之间并不会共享数据。

1.2 类的加载

​ 当程序主动使用某个类时，如果该类还未被加载到内存中，则系统会通过加载、链接、初始化三个步骤将该类加载到内存中。正常情况下，JVM 会连续完成这三个步骤，所以有时候也把这三个步骤统称为类加载或类初始化。

​ 实际上，类加载指的是将类的 .class 文件读入内存，并为之创建一个java.lang.Class对象，也就是说，当程序中使用任何类时，系统都会为之创建一个java.lang.Class对象。

​ 类的加载由类加载器完成，类加载器通常由JVM提供，除此之外，开发者可以通过继承 ClassLoader 类来创建自己的类加载器。类加载器是所有程序运行的基础。（将二进制文件加载进内存形成大 Class 对象）

通过使用不同的类加载器，可以从不同来源加载类的二进制数据。通常有如下几种来源：

  • 从本地文件中加载 class 文件。
  • 从 JAR 包加载 class 文件。如 JDBC 编程中用到的数据库驱动类就放在 JAR 文件中，JVM 可以直接从 JAR 文件中加载该 class 文件。
  • 通过网络加载 class 文件。

​ 类加载器可以无需等到“首次使用”该类时才加载该类，Java虚拟机允许系统预先的加载某些类。

1.3 类的链接

​ 当类被加载进内存后，系统就会为之生一个对应的 Class 对象，接着会进入链接阶段，连接阶段负责把类的二进制数据合并到 JRE中。类连接又可以分为如下3个阶段：

  1. 验证：验证阶段用于检验被加载的类是否有正确的内部结构，是否和其它类协调一致。
  2. 准备：类准备阶段则负责为类的静态变量分配内存，并设置默认初始值。
  3. 解析：将类的二进制数据中的符号引用替换成直接引用。

1.4 类的初始化

​ 在类的初始化阶段，主要就是对类中的静态变量进行显式初始化。在Java类中对静态变量指定初始化值的方式有2种：

  1. 声明静态变量时指定初始值。
  2. 使用静态代码块为静态变量指定初始值。
  3. 如果不指定静态变量的值，将会采用对应数据类型的默认零值作为初始值。

JVM初始化一个类包含如下几个步骤：

  1. 假设这个类还没有被加载和链接，则程序加载并链接该类。

  2. 假设该类中的直接父类还没有被初始化，则先初始化其直接父类。以此类推，所以 JVM 最先初始化的总是java.lang.Object类。

  3. 假设类中有初始化语句，则系统一次执行这些初始化语句。（如赋初始值等）

1.4.1 类的初始化时机

当Java程序首次通过如下方式来使用某个类（或接口）时，系统就会初始化该类（或接口）。

  1. 创建类的实例。包括：
    • 使用new创建实例。
    • 通过反射来创建实例
    • 通过反序列化来创建实例
  2. 调用某个类的静态方法。
  3. 访问某个类（或接口）的静态变量，或为该静态变量赋值。
  4. 使用反射方式来创建某个类（或接口）对应的java.lang.Class对象。
  5. 初始化某个类的子类。

2 初识反射

2.1 编译时类型 & 运行时类型

​ Java 程序中的很多对象在运行时都会出现两种类型：编译时类型 和 运行时类型。例如代码 Person p = new Student()，这行代码将会生成一个 p 变量，该变量在编译时类型为 Person ，运行时类型为 Student。除此之外，还有更极端的情况，程序在运行时接收到外部传入的一个对象，该对象的编译时类型是 Object ，但程序运行时又需要调用对象中某些具体实现的方法。

​ 为了解决这些问题，程序需要在运行时发现对象的真实信息。解决该问题的思路有两种：

  • 第一种思路就是让程序在编译时就能知道对象的具体信息，这时再交到内存中运行时运行时类型就是具体的实现类类型。
  • 第二种思路就是编译时根本无法知道该对象的具体信息，程序只能在运行时来获取对象的具体信息，这就必须使用反射。

2.2 什么是反射

定义：

  • Reflection（反射）是被视为动态语言的关键，
  • 反射机制允许程序在运行时借助于 Reflection API 取得任何类的内部信息，并能直接操作任意对象的内部属性及方法。

说明：

Java中的反射就是：在JVM处于运行状态时，

  • 对于任意一个类，都能知道这个类的所有属性和方法；
  • 对于任意一个对象，都能调用它的属性和方法

这种动态获取类的信息和动态调用对象方法的功能称为Java语言的反射机制。

之所以强调属性和方法是因为这是开发者最为关注的2个部分，实际上通过反射，不仅仅可以获取类的属性、方法，还可以获取类的父类、接口、包等信息。

静态调用和动态调用：

  • 静态调用：在.java文件中显式调用对象方法或获取类的信息。
  • 动态调用：在JVM运行时期，通过反射机制调用类/对象中的方法或获取类的信息。
  • 动态调用的好处：将编译期异常转移到运行期异常，降低耦合度。

3.2 Class 对象阶段[反射]

  1. Person.class 字节码文件会被 ClassLoader 加载到内存中，形成一个 Class 对象，该对象中保存了 Person 类的所有信息。

反射：将类的各个组成部分封装为一个个对象。

比如：

  • 将Person类的每个成员变量封装为Field对象，并存入Field[]数组中。
  • 将Person类的每个普通方法封装为Method对象，并存入Method[]数组中。
  • 将Person类的每个构造方法封装为Constructor对象，并存入Constructor[]数组中。

这一阶段是在内存中进行的。

这一阶段 Person 的 Class 对象整体仍然是一个对象，但是其中的成员变量、成员方法、普通方法等也分别被封装成了一个个的对象。

与 Java 反射相关的类：

类名 说明
Class 代表类的实体，在运行的Java程序中表示类或接口
Field 代表类的属性
Method 代表类中的普通方法
Constructor 代表类中的构造方法

3.3 运行时阶段（Runtime）

  • 运行时阶段（Runtime）的主要功能就是：将 Class 类转换为实例对象。
  • 这一阶段也在内存中进行。
