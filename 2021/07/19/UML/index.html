UML；Unified Modeling Language；统一建模语言

UML 是一种用于软件系统分析和设计的语言工具，它用于帮助软件开发人员进行思考和记录思路的结果。

UML 图分类：

  1. 用例图
  2. 静态结构图：
    • 类图
    • 对象图
    • 包图
    • 组件图
    • 部署图
  3. 动态行为图：
    • 交互图（时序图与协作图）
    • 状态图
    • 活动图

概述

UML 本身是一套符号的规定，就像数学符号和化学符号一样 ，这些符号用于描述软件模型中的各个元素和它们之间的关系，比如

  • 【元素】：类、接口
  • 【关系】：实现、泛化、依赖、组合、聚合等。

如下图所示：

UML 类图用于描述系统中的类（对象）本身的组成和类（对象）之间的各种静态关系。

类之间的关系（6种）：

  1. 依赖：
    • 只要是在类中用到了对方，那么他们之间就存在依赖关系。如果没有对方，连编绎都通过不了。
    • 使用带箭头的虚线表示依赖关系，如 --->。
  2. 泛化（继承）
    • 泛化关系实际上就是继承关系，它也是一种依赖关系。
    • 使用直线 + 空心三角形表示泛化关系。
  3. 实现
    • 实现关系实际上就是 A 类实现 B 接口， 也是一种依赖关系。
    • 使用 虚线 + 实心三角形 表示实现关系。
  4. 关联
  5. 聚合
  6. 组合

这些类之间的关系并不是相互独立的，它们可能存在重合的部分，即某两个类之间既存在 A 关系，又存在 B 关系。

依赖（Dependence）

只要是在类中用到了对方，那么他们之间就存在依赖关系。如果没有对方，连编绎都通过不了。

小结：类中用到了对方（类与类之间的关系），

  1. 是类的成员属性；
  2. 是类的方法中的参数；
  3. 是类的方法中的返回值；
  4. 在类中的方法内使用到了。

类图

  • PersonServiceBean 依赖 PersonDao、IDCard、Person、Department

泛化（继承）（generalization）

泛化关系实际上就是继承关系，它也是一种依赖关系。

小结

  1. 泛化关系实际上就是继承关系。
  2. 如果 A 类继承了 B 类，我们就说 A 和 B 存在泛化关系。

类图

Son 继承自 Father，所以说 Father 是 Son 的泛化。所谓泛化，可以理解为一种抽象，泛化就是说父类比子类具有更高层次的抽象，更具有普适性。

实现（Implementation）

实现关系实际上就是 A 类实现 B 接口， 也是一种依赖关系。

类图

PersonServiceBean 实现了 PersonService

关联（Association）

关联关系实际上就是类与类之间的联系，也是依赖关系的特例。

  • 关联具有导航性，即区分单向关联关系或双向关联关系。
  • 关系具有多重性，即划分为 一对一、一对多、多对多。

类图

聚合（Aggregation ）

聚合关系表示的是整体和部分的关系，整体与部分可以分开。聚合关系是关联关系的特例，所以它具有关联的导航性与多重性。

类图

如：一台电脑由显示器（monitor）、鼠标（mouse）等组成；组成电脑的各个配件是可以从电脑上分离出来的，就叫做聚合。

组合（Composition）

组合关系也是整体与部分的关系，但是整体与部分不可以分开。

如果我们认为 Mouse、Monitor 和 Computer 是不可分离的，则升级为组合关系。[Mouse 和 Monitor 对象随着 Computer 对象的创建而创建，销毁而销毁]

再看一个案例：在程序中我们定义实体：Person 与 IDCard 、Head。那么 Head 和 Person 就是组合，IDCard 和 Person 就是聚合。
