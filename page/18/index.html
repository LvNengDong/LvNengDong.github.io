函数式接口

包含且仅包含一个抽象方法的接口称为函数式接口。

  • 不是规定接口中只能有一个方法，非抽象方法的个数是不加限制的。
  • 当然在 JDK8 中接口中只能含有抽象方法和静态方法，所以这里的非抽象方法在 JDK8 中就是指静态方法，在 JDK9 中接口中还可以声明 private 方法。

@FunctionalInterface

  • 我们通过在任一接口上使用 @FunctionalInterface 注解来判断当接口是否为函数式接口。如果某个接口是一个函数式接口，在 JavaDoc 中也会包含一条声明加以说明。

Lambda表达式

Lambda 表达式就是对接口中抽象方法的实现代码。

Lambda 表达式是一种编程方式。

原来的匿名内部类 -->> Lambda 表达式

Lambda 表达式解决了匿名内部类代码语法冗余的现象。

Lambda 表达式要求接口必须是函数式接口，即接口中只有一个抽象方法，所以在重写接口中的抽象方法时不会产生歧义，因此方法名就可以省略。

Lambda 是一个匿名函数，我们可以把 Lambda 表达式理解为一段可以传递的代码(将代码看成一个整体，像变量一样进行传递)。使用 Lambda 表达式可以写出更简洁、更灵活的代码。

Lambda 表达式整体可以视为一个看起来很长的变量，可以作为方法中的参数传递。

Lambda表达式，也可以称为闭包，是 Java8 的新特性。

Lambda 允许把函数作为一个方法的参数(函数作为参数进行传递)。

实现方式口诀：

拷贝小括号，写死右箭头，落地大括号。

  1. 拷贝小括号：将接口中抽象方法的参数列表全部拷贝过来；
  2. 写死右箭头；
  3. 落地大括号：具体的落地实现放在大括号中

函数式编程

函数

  • f(x) = kx + b，f(x) 是要求得的计算结果， kx+b 是计算过程。

函数式编程

  • 就是让一个指定类型的变量表示 f(x)，用 Lambda 表达式表示计算过程的编程。

Lambda表达式语法

格式

Lambda 表达式用于实现接口中的抽象方法，那么显然它应该包含 参数列表、方法体、返回值类型 三个部分。

Java8 中引入了一个新的操作符 ->，它将 Lambda 表达式分为两个部分：

  • 左侧：Lambda 表达式的参数列表；对应接口中抽象方法的参数列表；
  • 右侧：Lambda 表达式的方法实现体；对应接口中抽象方法的实现类的方法体。
  • 返回值类型根据方法体中的返回值自行推断。

Lambda表达式对接口的要求

  • 需要函数式接口的支持。即 @FunctionalInterface 接口

  • 因为函数式接口中有且仅有一个抽象方法，所以使用 Lambda 表达式实现的抽象方法是唯一确定的。

Lambda语法优化

  • 可选的类型声明：
    • 不需要声明参数类型，因为 JVM 编译器可通过上下文推断出数据类型(根据接口中的抽象方法推断)。
    • 可以不写参数类型，但是必须所有参数都不写参数类型。
  • 可选的参数圆括号：
    • 有且仅有一个入参时可以省略参数小括号；
    • 零个或多个入参都不能省略小括号。
  • 可选的大括号：
    • 如果方法体只包含了一条语句，可以省略大括号。
  • 可选的返回关键字：
    • 如果方法体中唯一的一条语句是 return 语句，可以省略 return 关键字和大括号。
    • return关键字和大括号要么全都省略，要么全都不省略。

方法引用

概念

在前面的学习中，Lambda 表达式中的方法的具体实现都是我们自己写的，但是究其本质，Lambda 表达式就是一个函数（方法），所以 Lambda 表达式中的方法实现并非一定要我们自己来实现，我们可以直接引用 Java 中现成的方法实现作为 Lambda 表达式的方法体，这就叫做方法引用。

可以理解为方法引用是 Lambda 表达式的另外一种表现形式。

语法格式

  • 对象::实例方法名
  • 类名::静态方法名
  • 类名::实例方法名

eg.1

Consumer<T> 是一个消费型接口，接口中的抽象方法是 void accept(T t)，一个参数，无返回值。

  • System.out 实际上指向了一个 PrintStream 类型的实例，所以采用 对象::实例方法名 的格式。

使用要求：

  • Consumer<T> 接口中的抽象方法 accept() 有一个参数无返回值，所以对应的 println() 方法也要是一个参数无返回值的方法。
  • 即：引用方法的参数列表和返回值类型 要和 函数式接口中抽象方法的参数列表和返回值的个数与类型 要一一对应，相当于用引用方法实现了接口中的抽象方法。

构造器引用

概念：

构造器引用就是将构造方法作为 Lambda 表达式的方法体。

我们需要声明一个函数式接口，将接口中的抽象方法作为对象生成器方法，调用时返回一个新创建的对象。

语法：类名::new

在调用接口中的抽象方法时，通过 Lambda 表达式的构造器引用(类名::new)来执行构造器方法创建对象。

注意：虽然构造方法无返回值，但是默认返回一个对象实例，所以其对应的函数式接口应该是有返回值的。

  • 存在的问题？

    Employee 的构造方法不只一个，当使用构造器引用时，具体调用的是哪一个构造方法呢？

  • 答案：

    引用的构造器方法必须和函数式接口中的抽象方法的参数列表一致。在本例中，Supplier<T> 接口中的抽象方法是 T get()，无参数，所以引用的构造方法必然是无参构造方法。如果使用 Function<T, R> 接口中的 R apply(T t) 抽象方法，那么引用的构造器就必须包含一个参数！

Tip:需要调用的构造器方法的参数列表要与函数式接口中抽象方法的参数列表保持一致！

数组引用

概念

数组引用就是把创建数组的方法封装到 Lambda 表达式的方法体中。

我们需要声明一个函数式接口，该接口作为数组对象的生成器，当调用接口中的抽象方法时，实际就会调用 Lambda 方法体中创建数组的方法创建一个数组并返回。

语法格式：Type::new

Stream API

概念

Stream：流

Stream 是 Java8 中处理集合的关键抽象概念，它可以指定你希望对集合进行的操作，可以执行非常复杂的查找、过滤和映射数据等操作。使用 Stream API 对集合数据进行操作，就类似于使用 SQL 执行数据库查询。 也可以使用 Stream API 来并行执行操作。 简而言之，Stream API 提供了一种高效且易于使用的处理集合数据的方式。

Stream 用于操作数据源（集合、数组等）所生成的元素序列。

包：java.util.stream.*

Tip：

  • Stream 自己不会存储元素。
  • Stream 不会改变源对象。相反，它会返回一个持有结果的新的 Stream 对象。
  • Stream 操作是延迟执行的，这意味着它们会等到需要结果的时候才执行。

Stream 操作的三个步骤

  1. 创建 Stream 对象
    • 一个数据源（如：集合、数组）
    • 获取一个流对象
  2. 中间操作
    • 一个中间操作链，对包含数据源数据的流对象进行处理
  3. 终止操作（终端操作）
    • 一个终止操作 ，执行中间操作链，并产生结果。

创建Stream对象

创建 Stream 对象的 4 种方法

  1. Java8 中的 Collection 接口被扩展，提供了两个获取流的方法(即所有的 Collection 子类对象都可以调用这两个方法获取 Stream 实例)。
  2. 通过 Arrays 类中的的静态方法 stream() 可以获取数组流。
  3. 通过 Stream 类中的静态方法 of() 获取流对象
  4. 通过 Stream 类中的静态方法 iterate 或 generate 创建无限流。
    • 无限流就是如果不加限制 ，将会一直生成数据的流。

练习

并行流与串行流

并行流就是把一个内容分成多个数据块，并用不同的线程分别处理每个数据块的流。

Java8 中将并行进行了优化，我们可以很容易的对数据进行并行操作。Stream API 可以声明性的通过parallel()与sequential()在并行流和串行流之间进行切换。

接口中的默认方法与静态方法

  • Java8 之前，接口中只允许定义抽象方法（只有方法的定义，而无方法的实现）；、
  • 从 Java8 开始，接口中可以存在方法的实现，但是这个方法必须被 default 关键字修饰。
  • 从 Java8 开始，接口中也可以定义静态方法。

配置Swagger要扫描的接口

修改配置类（新增）

Docket.select

对象 方法 说明
Docket select() Docket对象中有一个select()方法，返回值是一个ApiSelectorBuilder对象，
用于配置Swagger要扫描的接口
ApiSelectorBuilder build() ApiSelectorBuilder对象中有一个build()方法,返回值是一个Docket对象

我们可以在select()和build()方法之间添加扫描条件/过滤条件等，自定义配置要扫描的接口。

对象 方法 说明
ApiSelectorBuilder .apis(RequestHandlerSelectors.basePackage()) 指定要扫描的包
同上 .apis(RequestHandlerSelectors.any()) 扫描全部
同上 .apis(RequestHandlerSelectors.none()) 全都不扫描
同上 .apis(RequestHandlerSelectors.withClassAnnotation(RestController.class)) 扫描类上的注解，参数是一个注解的反射对象。
效果：只有加了特定注解的类才能被扫描到。
同上 .apis(RequestHandlerSelectors.withMethodAnnotation(GetMapping.class)) 扫描方法上的注解，参数是一个注解的反射对象
同上 .paths(PathSelectors.ant("com.nengdong.swaggerdemo")) 过滤，被配置到过滤中的路径不能被扫描到（过滤的优先级高于扫描）

.select() ...... .build()包含扫描/过滤条件，是一个整体，两个方法之间只能使用.apis()和.paths()方法，其它方法需要在这个整体之外。

  • build() 方法采用了建造者模式，最后的 build 才是真正创建对象的动作。

MyBatis的XML映射器

1、自动映射

MyBatis的自动映射resultType可以自动映射同名的查询结果。但是如果遇到复杂的场景，还需要构建一个结果映射resultMap。实际上，这两种映射策略可以混合使用。

当自动映射resultType生效时，MyBatis会获取SQL语句执行结果中返回的列名并在Domain类中查找相同名字的属性（忽略大小写）。也就是说，如果发现了ID列和id属性，MyBatis会将列ID的值赋给id属性。

通常数据库列使用大写字母组成的单词命名，单词间用下划线分隔；而Java属性一般遵循驼峰命名法规定，为了在这两种命名方式之间启用自动映射，需要将==mapUnderscoreToCamelCase==设置为true。这种情况下，如果发现了USER_NAME列和userName属性，MyBatis会将列USER_NAME的值赋给userName属性。

甚至在提供了结果映射后，自动映射也能工作。处理的优先级是：手动映射 > 自动映射。

在下面的例子中，user_password列将根据resultMap进行映射，而user_id和user_name将根据resultType自动映射。

有3种自动映射等级：

  • ==NONE==[禁用]：禁用自动映射。仅对手动映射（ResultMap）的属性进行映射
  • ==PARTIAL==[局部]：如果有ResultMap则使用ResultMap，如果ResultMap只映射了部分字段，则剩余字段会使用ResultType进行映射。**默认值是PARTIAL**。
