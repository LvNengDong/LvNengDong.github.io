跨域问题常见的解决方案

Posted on 2021-04-06 Edited on 2021-04-11 In 开发日记

什么是跨域问题

跨域问题是浏览器对于 Ajax 请求的一种安全限制：一个页面发起的ajax请求，只能是与当前页域名相同的路径，这能有效的阻止跨站攻击。

以下情况都属于跨域：

跨域原因说明 示例
域名不同 www.jd.com 与 www.taobao.com
域名相同，端口不同 www.jd.com:8080 与 www.jd.com:8081
二级域名不同 item.jd.com 与 miaosha.jd.com

如果域名和端口都相同，但是请求路径不同，不属于跨域，如：

www.jd.com/item

www.jd.com/goods

而我们刚才是从manage.leyou.com去访问api.leyou.com，这属于二级域名不同，跨域了。

跨域问题常见的解决方案

  1. nginx 反向代理

利用 nginx 把跨域反向代理为不跨域，支持各种请求方式

缺点：需要在nginx进行额外配置，语义不清晰

  1. CORS

规范化的跨域请求解决方案，安全可靠。

优势：

  • 在服务端进行控制是否允许跨域，可自定义规则
  • 支持各种请求方式

缺点：

  • 会产生额外的请求

CORS 解决跨域

什么是 CORS

CORS 是一个 W3C 标准，全称是”跨域资源共享”（Cross-origin resource sharing）。

它允许浏览器向跨源服务器，发出 XMLHttpRequest 请求，从而克服了AJAX只能同源使用的限制。

CORS需要浏览器和服务器同时支持。

  • 浏览器端：

    目前，所有浏览器都支持该功能（IE10以下不行）。整个CORS通信过程，都是浏览器自动完成，不需要用户参与。

  • 服务端：

    CORS通信与AJAX没有任何差别，因此你不需要改变以前的业务逻辑。只不过，浏览器会在请求中携带一些头信息，我们需要以此判断是否允许其跨域，然后在响应头中加入一些信息即可。这一般通过过滤器完成即可。

CORS 原理

浏览器会将 Ajax 请求分为两类，其处理方案略有差异：简单请求、特殊请求。

简单请求

只要同时满足以下两大条件，就属于简单请求。：

（1) 请求方法是以下三种方法之一：

  • HEAD
  • GET
  • POST

（2）HTTP的头信息不超出以下几种字段：

  • Accept
  • Accept-Language
  • Content-Language
  • Last-Event-ID
  • Content-Type：只限于三个值application/x-www-form-urlencoded、multipart/form-data、text/plain

当浏览器发现发起的ajax请求是简单请求时，会在请求头中携带一个字段：Origin.【发起请求的域名】

Origin中会指出当前请求属于哪个域（协议+域名+端口）。服务会根据这个值决定是否允许其跨域。

如果服务器允许跨域，需要在返回的响应头中携带下面信息：

  • Access-Control-Allow-Origin：可接受的域，是一个具体域名或者*（代表任意域名）
  • Access-Control-Allow-Credentials：是否允许携带cookie，默认情况下，cors不会携带cookie，除非这个值是true

有关cookie：

要想操作cookie，需要满足3个条件：

  • 服务的响应头中需要携带Access-Control-Allow-Credentials并且为true。
  • 浏览器发起ajax需要指定withCredentials 为true
  • 响应头中的Access-Control-Allow-Origin一定不能为*，必须是指定的域名

特殊请求

不符合简单请求的条件，会被浏览器判定为特殊请求,，例如请求方式为PUT。

预检请求

特殊请求会在正式通信之前，增加一次HTTP查询请求，称为”预检”请求（preflight）。

浏览器先询问服务器，当前网页所在的域名是否在服务器的许可名单之中，以及可以使用哪些HTTP动词和头信息字段。只有得到肯定答复，浏览器才会发出正式的XMLHttpRequest请求，否则就报错。

一个“预检”请求的样板：

与简单请求相比，除了Origin以外，多了两个头：

  • Access-Control-Request-Method：接下来会用到的请求方式，比如PUT
  • Access-Control-Request-Headers：会额外用到的头信息

预检请求的响应

服务的收到预检请求，如果许可跨域，会发出响应：

除了Access-Control-Allow-Origin和Access-Control-Allow-Credentials以外，这里又额外多出3个头：

  • Access-Control-Allow-Methods：允许访问的方式
  • Access-Control-Allow-Headers：允许携带的头
  • Access-Control-Max-Age：本次许可的有效时长，单位是秒，过期之前的ajax请求就无需再次进行预检了

如果浏览器得到上述响应，则认定为可以跨域，后续就跟简单请求的处理是一样的了。

跨域实现

虽然原理比较复杂，但是前面说过：

  • 浏览器端都有浏览器自动完成，我们无需操心
  • 服务端可以通过拦截器统一实现，不必每次都去进行跨域判定的编写。

事实上，SpringMVC已经帮我们写好了CORS的跨域过滤器：CorsFilter ,内部已经实现了刚才所讲的判定逻辑，我们直接用就好了。

通过 @CrossOrigin 注解实现

通过 @CrossOrigin 注解实现跨域请求非常简单，只需要在要访问的 Controller 上加上 @CrossOrigin 注解即可！

缺点是不能全局生效，每个要跨域的 Controller 都需要添加 @CrossOrigin 注解！

通过全局配置文件实现

  1. 编写一个配置类，并且注册 CorsFilter

常见的错误

常见错误1：

@CrossOrigin注解 和 全局配置文件 不能同时存在，因为这两种方式都能起到允许跨域的效果，如果同时使用就会发生冲突！

常见错误2

错误信息：

Cross origin requests are only supported for protocol schemes: http, data, chrome, chrome-extension, chrome-untrusted, https

解释说明：跨域请求只支持以下协议：http, data, chrome, chrome-extension, chrome-untrusted, https。

分析：

toString()、(String)、String.valueOf(Object)的区别

在 Java 项目的实际开发和应用中，常常需要用到将对象转为 String 这一基本功能。

常用的方法有：

  • Object.toString()
  • (String)Object
  • String.valueOf(Object)

1 采用 Object.toString() 方法

  • 因为 toString() 方法是 java.lang.Object 类里的 public 方法，所以任何 Java 对象都可以调用此方法。
  • 但在使用时要注意，必须保证 Object 不是 null 值，否则将抛出 NullPointerException 异常。
  • 采用这种方法时，通常派生类会覆盖 Object 里的 toString() 方法。

2 采用类型转换 (String)Object 方法

  • 使用这种方法时，需要注意的是类型必须能转成String类型。因为 Object 类型的对象在转成 String 对象时语法检查并不会报错，这将可能导致潜在的错误存在。所以再转换时最好用 instanceof() 方法做个类型检查，以判断是否可以转换。否则容易抛出 CalssCastException 异常。

  • 此外，因 null 值可以强制转换为任何 Java 类类型，所以 (String)null 也是合法的。

    null 值经过转换后还是 null。注意是 空值null 而不是 字符串null ！！！

3 采用 String.valueOf(Object) 方法

String.valueOf(Object) 的基础是 Object.toString() ，但它与 Object.toString() 又有所不同。在前面方法1的分析中提到，使用toString(Object) 方法时需保证 Object 不为 null 。但在采用 String.valueOf(Object) 方法时，将不用担心 Object 是否为 null 这一问题。

多表联查(三张或三张以上)

SQL 的执行顺序

总结：

  • 所有的查询语句都是从 from 开始执行的
  • 在执行过程中，每个步骤都会为下一步骤生成一个虚拟表，这个虚拟表将作为下一个执行步骤的输入！

分析：

  • step1：首先对 from 子句中的前两张表（可能查询大于两张表）执行一个笛卡尔乘积，此时生成虚拟表 vt1（Virtual Table）！

  • step2：接下来就是使用 on 筛选器。on 中的逻辑表达式将应用到 vt1 中的每一行，筛选出满足 on 逻辑表达式的行，生成虚拟表 vt2 ！

  • step3：如果是 outer join 那么这一步就将添加外部行，如果是 left outer jion 就把左表在第二步中过滤的添加进来，如果是 right outer join 那么就将右表在第二步中过滤掉的行添加进来，这样生成虚拟表 vt3 。

WebPack 概述

什么是 SPA

SPA（Single Page Application）：单页应用。即整个应用系统只有一个 HTML 页面，剩余的一切页面内容都是通过在这个 HTML 页面中引入 Vue 组件来实现的。 Vue 中的局部组件其实就是一些普通的 Js 对象（不过一般我们会将每一个 Js 对象都单独保存到一个 Js 文件中）。

不过一个完整的前端项目中除了 Js ，还应该有 Css、images 等资源文件，这些东西在打包、构建的过程中，人工操作非常麻烦，因此就会有一些工具来帮助搭建前端项目。例如：WebPack、vue-cli 等。

这些全部的资源被打包后，一般会形成一个或少数几个打包文件（一般是 Js 文件），我们只需要在那个唯一的 HTML 页面中引入打包形成的 Js 文件，就相当于引入了整个前端项目的所有资源文件。

什么是 WebPack

Webpack 是一个前端资源的打包工具，它可以将 js、image、css 等资源当成一个模块进行打包。

中文官方网站：https://www.webpackjs.com/

WebPack 的作用

  1. 打包

在开发前端项目时，一个项目中往往会包含成千上万个小的静态文件，如 images、css、html、js 等，因为这些小的文件彼此之间经常会相互调用，如果项目不打包就直接部署的话，用户在请求单页面时就会衍生出成百上千次请求，显然对服务器的压力就非常大。

经过 WebPack 打包后，可以将这些小的文件合并成较大的文件，减少单页面内衍生的请求次数。

  1. 对代码进行转换

为了简化开发，前端语言在发展过程衍生出了许多编程语言，如 ts、sass、hbs、cjs 等，而浏览器只能识别 htm、css、js 和 静态资源等。所以如果我们直接拿一些高级编程语言让浏览器解析，显然是不会成功的。

而 WebPack 第二大作用就是能对目前的主流编程语言进行编译，将浏览器不能识别的编程语言编译成浏览器能识别的编程语言。

  1. 对语法进行降级

目前主流的浏览器都是不兼容 ES6 语法的。如果某个应用是使用 ES6 语法开发的，显然不能直接运行在浏览器上。

而使用 WebPack 就可以将 ES6 语法进行转换编译为 ES5 或更老的版本，以兼容不同内核的浏览器。

  1. 对代码进行混淆，提高代码的安全性

WebPack 在将代码打包的同时进行混淆，提高代码的安全性。

Webpack 核心概念

入口(entry)

一个完整前端项目包含很多文件，打包就是要把项目运行需要的文件合并成一个或几个。所以通过 Webpack 的入口一定要能找到项目中其它所有的资源文件。

Webpack 打包的起点，可以有一个或多个，一般是一个 Js文件，这个 Js 文件必须把所有要打包的其它文件直接或间接的引用起来。

Webpack 会从起点文件开始，寻找起点直接或间接依赖的其它所有的依赖，包括 Js、Css、图片资源等，作为将来打包的原始数据。

输出(output)

出口一般包含两个属性：path 和 filename。用来告诉 Webpack 打包的目标文件夹，以及文件的名称。目的地也可以有多个。

加载器（loader）

Webpack 本身只识别 Js 文件，如果要加载非 Js 文件，必须指定一些额外的加载器（loader），例如 css-loader。然后将这些文件转为Webpack 能处理的有效模块，最后利用 Webpack 的打包能力去处理。

插件(plugins)

插件可以扩展 Webpack 的功能，让 Webpack 不仅仅是完成打包，甚至各种更复杂的功能，或者是对打包功能进行优化、压缩，提高效率。

SpringBoot与缓存

什么缓存

缓存是每一个系统都应该考虑的功能，用来加速系统的访问和提升系统的性能。

  • 对于一些我们需要经常访问的高频数据，如果每次访问都从数据库中查询的话，这样的操作耗时太大了。
  • 为了解决这个问题，我们可以引入缓存中间件，将需经常访问的数据保存在缓存中间件中，在进行数据访问时如果缓存中含有这些数据，就不再去数据库中查询了，而是直接从缓存获取数据，这样可以大大降低访问时间！ 如果缓存中没有这些数据，就从数据库中查询这些数据，并将查到的数据再放缓存中！
  • 对于一些临时性数据，比如手机的验证码（假设3分钟内有效），如果把所有的验证码数据都存入到数据库中，对于数据库的压力是很大的。我们可以把这些临时性的数据也存到缓存中，从缓存中存取临时性的数据。

为了统一缓存的开发规范和提升系统的可扩展性，就有了 JSR-107 缓存规范！

总结：

缓存中适合保存的数据：

  1. 需要经常访问的数据
  2. 临时性数据

JSR-107 缓存规范

JSR-107缓存规范定义了5个核心接口，分别是：

  • CachingProvider 缓存提供者
  • CacheManager 缓存管理器
  • Cache 缓存（组件）
  • Entry 缓存记录（K，V）
  • Expiry Entry 的有效期

  1. CachingProvider

    • 管理多个CacheManager。
    • 一个应用可以创建多个多个CachingProvider。

  2. CacheManager

    • 管理多个唯一命名的Cache。
    • 这些Cache存在于CacheManager的上下文中。一个CacheManager仅被一个CachingProvider所拥有。

  3. Cache

    • 以 <K, V> 的数据结构临时存储以 Key 为索引的值。
    • 一个Cache仅被一个CacheManager所拥有。

  4. Entry

    • 是一个存储在 Cache 中的 key-value 对。

  5. Expiry

    每一个存储在 Cache 中的条目有一个定义的有效期。一旦超过这个时间，条目为过期的状态。一旦过期，条目将不可访问、更新和删除。缓存有效期可以通过 ExpiryPolicy 设置。

Spring 缓存抽象

Spring 中的缓存API & 缓存注解

  • **Spring 简化了 JSR-107 规范，只保留了 Cache 和 CacheManager 接口来统一不同的缓存技术！ ****
  • 并支持使用注解简化我们开发！

EasyExcel

EasyExcel 是一个基于 Java 的简单、省内存的读写 Excel 的开源项目。在尽可能节约内存的情况下支持读写百M的 Excel 。

SVN 是『集中式版本控制系统』，版本库是放在中央服务器的，而干活的时候，用的都是自己的电脑，所以首先要从中央服务器哪里得到最新的版本，然后干活，干完后，需要把自己做完的活推送到中央服务器。

集中式版本控制系统的限制是必须联网才能工作，如果在局域网还可以，带宽够大，速度够快，如果在互联网下，如果网速慢的话，就郁闷了。

下图就是标准的集中式版本控制工具管理方式：

集中管理方式在一定程度上看到其他开发人员在干什么，而管理员也可以很轻松掌握每个人的开发权限。

但是相较于其优点而言，集中式版本控制工具缺点很明显：

  • 服务器单点故障
  • 容错性差

NOSQL

NoSQL（Not Only SQL）：”不仅仅是SQL”，是一种全新的数据库理念，泛指非关系型数据库。

NoSQL vs. SQL

关系型数据库的优势

  1. 复杂查询：
    • 可以用SQL语句方便的在一个表以及多个表之间做非常复杂的数据查询
  2. 事务支持
    • 事务支持使得对于安全性能要求很高的数据访问需求得以实现。

关系型数据库的缺点：

  1. 在处理超大规模和高并发的数据时速度慢！

非关系型数据库的优点：

  1. 查询大型规模数据集合时速度快！
    • NOSQL 数据库将数据存储在缓存中，关系型数据库将数据存储在硬盘中。自然NOSQL数据库查询速度要更快！）
  2. 存储数据的格式多样化。
    • NoSQL 中数据的存储格式有：<K, V> 格式、文档形式、图片形式等等。 不仅可以存储基础类型的数据，还可以存储对象或者是集合等各种格式，而数据库则只支持基础类型。）
  3. 扩展性好。
    • 关系型数据库有类似于 join 这样的多表查询机制，导致数据库一旦创建就很难扩展。
    • 而NoSQL是基于键值对存储数据的，数据之间没有耦合，所以非常容易水平扩展。

非关系型数据库的缺点：

  1. 没有标准的SQL（结构化查询语句）的支持。

    （没有一个统一的标准来规定 NoSQL 的语法。 也就是说，对于不同的 NoSQL 数据库，增删改查等语句不尽相同，会增加用户的学习和使用成本。）

  2. 不提供关系型数据库对事务的处理。

总结

  1. 关系型数据库与NOSQL数据库并非对立而是互补的关系。即通常情况下使用关系型数据库，在适合使用NOSQL的时候使用NOSQL数据库。
  2. 一般情况下，会将数据存储在关系型数据库中，在NoSQL数据库中存储关系型数据库中数据的备份。

主流的NoSQL产品

键值对<K, V>存储数据库
相关产品 Tokyo Cabinet、Tyrant、Redis、Voldemort、Berkeley DB
典型应用 内容缓存，主要用于处理大量数据的高访问负载
数据模型 键值对
优势 快速查询
劣势 存储的数据缺少结构化

列存储数据库
相关产品 Cassandra、Hbase、Riak
典型应用 分布式的文件系统
数据模型 以列簇式存储，将同一列数据存在一起
优势 查找速度快，可扩展性强，更容易进行分布式扩展
劣势 功能相对局限

文档型数据库
相关产品 CouchDB、MongoDB
典型应用 Web应用（与key-value类似）
数据模型 键值对
优势 数据结构要求不严格
劣势 查询性能不高，而且缺乏统一的查询语法

图形数据库
相关产品 Neo4J、InfoGrid、Infinite Graph
典型应用 社交网络
数据模型 图结构
优势 利用图结构的相关算法
劣势 需要对整个图做计算才能得出结果，不容易做分布式的集群方案
