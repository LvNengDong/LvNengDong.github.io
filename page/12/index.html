分治算法

分治算法介绍

​ 分治法是一种很重要的算法，顾名思义就是“分而治之”，即把一个复杂的问题分解成两个或更多相同或相似的子问题，再继续把子问题分成更小的子问题，直到最后子问题可以简单的直接求解，原问题的解即子问题的解的合并。

​ 分治思想是很多高效算法的基础，如排序算法（快速排序、归并排序），傅立叶变换（快速傅立叶变换）等。

分治算法的基本步骤

分治法在每一层递归上都有三个步骤：

  1. 分解：将原问题分解为若干个规模较小，相互独立，与原问题形式相同的子问题；
  2. 解决：若子问题规模较小而容易被解决则直接解，否则递归地解各个子问题；
  3. 合并：将各个子问题的解合并为原问题的解。

最优解问题

我们通过一个实际问题来学习动态规划。

题目：

下图共有 7 项任务，横轴表示每项任务的工作时间，红字表示完成每项任务可以得到的报酬。比如任务一需要在 1~4 时间段进行，任务二需要在 3~5 时间段进行。也就是说想要完成任务一就必须放弃任务二、三、五，因为这几个任务的时间段冲突了。现在的问题是想知道从 0~11 时间段内最高报酬的任务组合是什么。

分析：

动态规划就是【选/不选】的问题！！！

我们用 OPT(i) 函数表示包含第 i 个任务的情况下报酬最高的任务组合。【OPT：最优解】

比如选了任务 ⑧，肯定就不能再选任务 ⑥ 和 ⑦ 了。这时就会存在两种情况，到底是选择任务 ⑧ 后得到的总薪资最高还是不选任务⑧得到的总薪资最高呢？这就是 【选/不选】的问题。
$$
OPT(8) = max
\begin{cases}
V_8+OPT(5) &\text{// 选⑧就要放弃⑥和⑦}
\\
OPT(7) &\text{//不选⑧就相当于放弃⑧}
\end{cases}
$$

公式：

以此类推，可以得到一个公式，
$$
OPT(i) = max
\begin{cases}
V_i+OPT(prev(i)) &\text{// prev(i)表示在选择了第i个任务后剩余任务中的最优解下标}
\\
OPT(i-1) &\text{//不选}
\end{cases}
$$

    • Vi 表示第 i 个任务的薪资；

    • prev(i) 表示选择第 i 个任务后，剩下任务中的最优解下标。

    • 所以可以采用动态规划的算法，【自底向上】建立一个动态规划的数组，保存从 OPT(0)~OPT(n) 的所有值。

动态规划算法介绍

  1. 动态规划（Dynamic Programming）算法的核心思想是：将大问题划分为小问题进行解决，从而一步步获取最优解的处理算法。
  2. 动态规划算法与分治算法类似，其基本思想也是将待求解问题分解成若干个子问题，先求解子问题，然后从这些子问题的解得到原问题的解。
  3. 与分治法不同的是，适合于用动态规划求解的问题，经分解得到的子问题往往不是互相独立的。即下一个子问题的求解是建立在上一个子问题解的基础上得到的。
  4. 动态规划可以通过填表的方式来逐步推进，得到最优解.

MySQL 架构

1.1 MySQL逻辑架构

MySQL 可以在多种不同场景中应用并发挥良好作用。主要体现在存储引擎的架构上，插件式的存储引擎架构将查询处理和其它的系统任务以及数据的存储提取相分离。这种架构可以根据业务的需求和实际需要选择合适的存储引擎。其主要逻辑架构可分为四层，从上到下分别是：

  1. 连接层
  2. 服务层
  3. 引擎层
  4. 存储层

1.1.1 连接层

  • 连接层主要处理客户端与 MySQL 服务端的连接、授权认证、连接安全等问题。包含本地 socket 通信和大多数基于客户端/服务端工具实现的网络通信。
  • 主要完成一些类似于连接处理、授权认证、及相关的安全方案。同样在该层上可以实现基于 SSL 的安全链接。服务器也会为安全接入的每个客户端验证它所具有的操作权限。
  • 在该层上引入了线程池的概念，为通过认证安全接入的客户端提供线程。

1.1.2 服务层

名字 介绍
Management Serveices & Utilities 系统管理和控制工具
SQL Interface SQL 接口。接受用户的SQL 命令，并且返回用户需要查询的结果。
Parser 解析器。SQL 命令传递到解析器的时候会被解析器验证和解析
Optimizer 查询优化器。SQL 语句在查询之前会使用查询优化器对查询进行优化。
Cache 和Buffer 查询缓存。

1、Management Serveices & Utilities：系统管理控制工具

可以看做是类似于 Hadoop 中的 Zookeeper，或者是 SpringCloud 中的配置中心，主要作用是协调整管理个系统的各个模块。

2、SQL Interface：SQL 接口

类似于 MVC 三层架构中的 Controller 层，只是负责接收请求和返回最终的结果，具体的业务逻辑交给更底层的服务去做。比如 select ... from ... 就会调用 SQL Interface。

3、Parser：解析器

SQL Interface 中的 SQL 命令传递到解析器的时候会被解析器验证和解析。主要功能有：

  • 验证：对接收的 SQL 语句进行语义和语法的分析，如果在分解构成中遇到错误，那么就说明这个 SQL 语句是不合理的，验证失败，不继续网下执行，直接抛出错误。
  • 解析：如果验证成功，则根据语义和语法，将 SQL 语句划分为不同的操作类型，然后进行针对性的转发到后续步骤。比如查询语句就转发到查询对应的服务，更新操作就转发到更新对应的服务。

4、Optimizer：查询优化器

  • SQL 语句在查询之前会使用查询优化器对查询进行优化。就是优化客户端请求的 SQL 语句，根据客户端请求的 SQL 语句，和数据库中的一些统计信息，在一系列算法的基础上进行分析，得出一个最优的策略，告诉后面的程序如何执行这个 SQL 语句。（机器认为最优的数据检索方式，但 BDA 不一定认为是最优的）
  • 查询优化器就类似于指令重排，对于开发人员来说，写一条 SQL 可能自我感觉良好，性能天下第一，但是对于底层机器来说，它有自己的优化标准，经过查询优化器优化后的 SQL 语句的执行顺序并不一定与开发人员设计的 SQL 执行顺序相同。
  • 当客户端向 MySQL 服务器请求一条 SQL ，命令解析器模块完成请求分类，区别出是 SELECT 并转发给 MySQL_Query_Qptimizer 时， MySQL_Query_Qptimizer 首先会对整条 SQL 进行优化，处理掉一些常量表达式的预算，直接转换成常量值。并对 SQL 中的查询条件进行简化和转换，如去掉一些无用或显而易见的条件、结构调整等。

5、Cache和Buffer：查询缓存

如果查询缓存有命中的查询结果，查询语句就可以直接去查询缓存中取数据。这个缓存机制是由一系列小缓存组成的。比如表缓存，记录缓存，key 缓存，权限缓存等。

1.1.3 引擎层

  • 存储引擎主要负责数据存储和提取，存储引擎不会去解析 sql，不同存储引擎之间不会通讯，只会简单地响应上层服务器的请求。
  • 存储引擎真正的负责了 MySQL 中数据的存储和提取，服务器通过 API 与存储引擎进行通信。
  • 存储引擎接口模块可以说是 MySQL 数据库中最有特色的一点了。目前各种数据库产品中，基本上只有 MySQL 可以实现其底层数据存储引擎的插件式管理。这个模块实际上只是 一个抽象类，但正是因为它成功地将各种数据处理高度抽象化，才成就了今天 MySQL 可插拔存储引擎的特色。
  • 可插拔的存储引擎的实现，让 MySQL 在处理不同的业务逻辑时，可以根据实际应用场景选择效率尽可能高的存储引擎。这就是 MySQL 相较于其它同类数据库产品的最大优势。
  • 注意：存储引擎是基于表的，而不是数据库。

1.3 大致查询流程

MySQL 查询的流程大致是：

  1. MySQL 客户端通过协议与 MySQL 服务器建连接；
  2. 发送查询语句，先查询缓存，如果命中，直接返回结果；
  3. 否则进行语句解析，也就是说，在解析查询之前，服务器会先访问查询缓存（Query Cache）——它存储了 SQL 语句以及
    相应的查询结果集。
  4. 如果某个查询结果已经位于缓存中，服务器就不会再对 SQL 语句进行解析、优化、以及执行。它仅仅将缓存中的结果返回给用户即可，这将大大提高系统的性能。

主从复制（MySQL 集群）

一主多从

  • 主节点用来执行写操作，从节点用于执行读操作
  • 所有在主节点上执行的写操作，都会被写入主节点的日志文件 Binary_log 中；
  • 从节点通过 IO 读取到主节点的日志文件，复制到 Relay_log 中，然后由当前备机执行日志文件，即可完成从机数据的同步。

读写分离：

  • 因为 MySQL 只会将主节点的 Binary_log 文件同步到从节点，而不会反过来同步。因此也就引出了读写分离的问题。
  • 如果要保证主从节点间的数据一致，写操作就只能发生在主节点，而读操作可以在集群中的任意节点上进行。

分库分表

为什么要进行分库分表？

当单张表中的数据量过大时，整体的查询效率就会降低得非常明显。这时为了提升查询效率，就要将一个表中的数据分散到多个数据库的多个表中。

多大数据量需要进行分库分表？

阿里开发手册中建议：一个表的数据量超过 500W 条或者数据文件超过 2G，就要考虑进行分库分表了。

分库分表的方式

  • 分库分表包括分库和分表两个部分，而这两个部分统称为数据分片，其目的都是将数据拆分成不同的存储单元。
  • 另外，从拆分的角度上，可以划分为垂直分片和水平分片。

垂直分片

  • 按照业务来对数据进行分片，又称为纵向分片。它的核心理念就是专库专用。
  • 在拆分之前，一个数据库由多张数据表组成，每张表对应不同的业务，而拆分之后，则是按照业务将表归类，分布到不同的数据库或表中，从而将压力分散至不同的数据库或表。
  • 例如：下图将用户表和订单表垂直分片到不同的数据库。

垂直分片往往需要对架构和设计进行调整。通常来讲，是来不及应对业务需求快速变化的。而且，也无法真正地解决单点数据库的性能瓶颈。垂直分片可以缓解数据量和访问量带来的问题，但无法根治。如果垂直分片之后，表中的数据量依然超过单节点所能承受的阈值，则需要水平分片来进一步处理。

水平分片

  • 水平分片又称为横向分片。
  • 相对于垂直分片，它不再将数据根据业务逻辑分类，而是通过某几个字段，根据某种规则将数据分散到多个库或表中，每个分片仅包含数据的一部分。
  • 水平分片从理论上突破了单击数据量处理的瓶颈，并且扩展相对自由，是分库分表的标准解决方案

分库分表后，SQL语句的执行流程是怎样的？

分库分表后的执行流程

分布分表后，在执行查询语句时，由于查询语句中使用的是逻辑上的表，但是在分库分表后，逻辑上的表结构实际存储在多个库的多个表中。

  1. 在执行 SQL 语句时，首先依然需要解析器解析和查询优化；
  2. 但是在查询时，由于数据保存在多张表中，所以需要进行 SQL 路由，根据查询条件去不同库的表中查询；
  3. 这时候需要使用到真实的库名称和表名称，所以会有一步 SQL 改写的过程，将逻辑上的 SQL 语句改写成一条或多条真正要去执行查询的 SQL 语句，然后再执行查询；
  4. 得到查询结果后，可能查询结果来自多个库的多张表，所以还需要将这多个结果进行归并，最后才能得到一个唯一的结果返回给客户端。

MySQL索引

Posted on 2021-07-23 Edited on 2021-10-21 In MySql

1. 是什么

定义：

  1. 索引是一种文件。
  2. 索引是一种数据结构。

文件：

索引是一个文件，它是要占据物理空间的。

  • 对于聚簇索引（如 InnoDB）而言，索引表中直接保存了对应的真实数据表中的所有的记录，所以 InnoDB 持久化后只会得到一个文件，即索引表数据和真实数据表数据都保存在一起。
  • 对于非聚簇索引（如 MyISAM）而言，索引表中保存了数据表中对应记录的引用（也就是主键），所以对于 MyISAM 而言，持久化后会得到两个文件，一个文件用于保存索引表的数据，另一个文件用于保存真实表的数据。

数据结构：

  • 索引是一种数据结构。可以理解为“排好序的可以实现快速查找的数据结构”，能够用于提高数据查找的效率。
  • 索引的具体实现可以有多种，常见的有 B-树、B+树、AVL树、Hash 等等。
  • 通过对某些列建立索引，可以避免对数据的全局扫描，以更少的查找次数快速找到要查询的数据。

小总结：

索引就相当于目录。为了方便查找书中的内容，通过对内容建立索引形成目录。MySQL 索引的建立对于 MySQL 的高效运行是很重要的，索引可以大大提高 MySQL 的检索速度。比如我们在查字典的时候，前面都有检索的拼音和偏旁、笔画等，然后找到对应字典页码，这样然后就打开字典的页数就可以知道我们要搜索的某一个key的全部值的信息了。

解释说明：

Before：

  • 如果没有索引，想要查询 tb_user 表中 name="张三" 的这条数据，我们需要挨个查询 tb_user 表中的所有数据，直到找到 name="张三" 的数据。当然这种查询存在随机性，可能第一个就直接查到了，也有可能直到最后一个才查到，平均时间复杂度为 O(n) 。

After：

  • 假设我们为 name 这一列建立了一个索引（假设索引的数据结构为 B+Tree），这时 name 列中的所有数据就会按照某种规则映射到 B+Tree 上，想要查询 name="张三" 的这条数据时，用 “张三” 对应的值与 B+Tree 第一层的多个范围空间进行对比，找到满足条件的范围进入下一层对比，这样只需要很少的几次比较，就能够定位到 “张三” 在 BTree 中的位置，再从索引直接关联到数据库中真实数据的内存地址，即完成了一次快速的查询。

索引会影响到两大功能的执行效率：

  1. 排序：ORDER BY
  2. 查找：WHERE；HAVING

解释：

  • 查找时，如果对应的字段建立了索引，就不会出现在某张表中挨个查找的情况，而是跳着查找，效率更高。
  • 排序时，因为索引结构本身就是有序的，所以对应的字段直接按照索引顺序正序或倒序排列即可。

2. 优缺点

优点：

  1. 索引能提高数据检索速率，对于一些常用于查询的字段，如果将其设置为索引，能大大地提高查询的速率。
    • 将频繁使用的字段就将其设置为索引，比不设索引时查询速度要快。
  2. 对索引列的数据进行排序，排序的成本将大大降低。

缺点：

  1. 需要维护额外的索引表的空间。索引也是一张表，该表保存了索引字段（主键天生就是索引字段），并指向实体表中的记录。所以索引也是要占用空间的。
  2. 在发生写操作时，创建和维护索引需要额外的时间。虽然索引大大提高了查询速度，但同时会降低写操作的执行效率。比如在对表进行 CRUD 操作时，不仅需要更新表中的数据，还要更新对应的索引表。
    • 对于 UPDATE 操作，记录变更后索引字段在索引表中的位置就可能发生变化，需要更新索引表。
    • 对于 INSERT 操作，添加记录后需要将新记录的索引字段加到索引表中。
    • DELETE 同上。

3. 索引分类

按应用层次分类

  • 单列索引：一个索引只包含单个列，一个表可以有多个单列索引。
  • 复合索引：即一个索引包含多个列。
  • 唯一索引：索引列的值必须唯一（即用于建立索引的列必须符合 UNIQUE），但允许有空值。【主键就是一种唯一索引】

复合索引

  • MySQL 可以使用多个字段同时建立一个索引，叫做复合索引。
  • 在复合索引中，如果想要命中索引，筛选条件需要严格按照建立索引时的字段顺序，否则无法命中索引。

具体原因：

  • MySQL 使用索引时需要筛选条件按照索引的创建方式严格有序。
  • 假设现在建立了 (id, name，age) 的联合索引，那么索引的排序为： 先按照 id 排序，如果 id 相同，再按照 name 排序；如果 name 的值也相同，再按照 age 排序，以此类推。
  • 在进行查询时，如果仅仅按照 id 查询，是可以命中索引的；按照 (id,name) 或者 (id,name.age) 的顺序查询，也是可以命中索引的。但是如果调换了顺序，比如将 name 或 age 提到 id 之前，按照 (name, id) 的顺序来查询，就无法命中索引了。

最左前缀原则

  • 最左前缀原则就是最左优先，在创建多列索引时，要根据业务需求，where 子句中使用最频繁的一列放在最左边。
  • MySQL 会一直向右匹配直到遇到范围查询(>、<、between、like)就停止匹配，比如 a = 1 and b = 2 and c > 3 and d = 4 如果建立 (a,b,c,d) 顺序的索引，d 是用不到索引的，如果建立 (a,b,d,c) 的索引则都可以用到，a,b,d的顺序可以任意调整。
  • = 和 in 可以乱序，比如 a = 1 and b = 2 and c = 3 建立 (a,b,c) 索引可以任意顺序，MySQL 的查询优化器会帮你优化成索引可以识别的形式。

按是否聚簇分类

  • 聚簇索引（聚集索引）：聚簇索引就是将索引数据和真实数据在物理上保存在一起，找到索引也就找到了数据。
  • 非聚簇索引：非聚簇索引就是真实数据和索引数据物理上不保存在一起，但是索引表中保存了指向真实数据的指针，要获取表中的真实数据有时还需要通过索引对应的地址空间到另一张专门保存数据的表中去找。

InnoDB 中的聚簇索引 & 非聚簇索引

在 InnoDB 里，

  • 索引 B+Tree 的叶子节点存储了整行数据的索引是主键索引，也被称之为聚簇索引，即将整行数据与索引放到了一块，找到索引也就找到了数据。
  • 而索引 B+Tree 的叶子节点存储了主键的值（在这里主键的值就是指向真实数据的指针）的是非主键索引，也被称之为非聚簇索引、二级索引。

聚簇索引与非聚簇索引的区别：

  • InnoDB 的聚簇索引其实就是在同一个结构中保存了 B+Tree 索引和数据行。
  • 非聚簇索引与聚簇索引的区别在于非聚簇索引的叶子节点不存储表中的数据，而是存储该列对应的主键（行号）。
  • 对于 InnoDB 来说，想要查找数据我们还需要根据主键再去聚簇索引中进行查找，这个再根据聚簇索引查找数据的过程，我们称为回表。
  • MyISAM 无论主键索引还是二级索引都是非聚簇索引，而 InnoDB 的主键索引是聚簇索引，二级索引是非聚簇索引。我们自己建的索引基本都是非聚簇索引。
  • 聚簇索引要求真实数据与索引存放在一起，所以一个表当中最好只有一个聚簇索引，而非聚簇索引可以有多个。如果想要一个表中能有多个聚簇索引，就需要给每一个聚簇索引分配一份真实表数据的副本。

MyISAM 只有非聚簇索引

  • MyISAM 使用的就是非聚簇索引，树的叶子节点上的 Data 不是数据本身，而是数据存放的地址。InnoDB 采用的是聚簇索引，树的叶子节点上的 Data 就是数据本身。
  • MySQL 的 MyISAM 引擎在建立索引时，会专门维护一张索引表，索引表中除了索引值外，还有一列用于存储真实数据的内存地址，这就叫做非聚簇索引。
  • 因为
  • InnoDB 中，如果表定义了 PrimaryKey，那么 PrimaryKey 就是聚簇索引。如果没有 PrimaryKey ，就会找第一个非空的 unique 列作用聚簇索引。否则，InnoDB 会创建一个隐藏的 row_id 作为聚簇索引。

非聚簇索引

在数据之外，数据库系统还维护着满足特定查找算法的数据结构，这些数据结构以某种方式指向数据（记录了真实真实数据的地址），在这些数据结构上我们可以实现高效的查找算法。这种数据结构，就是索引。

二叉排序树

  • 若它的左子树不为空，则左子树上所有关键字的值均小于根关键字的值；
  • 若它的右子树不为空，则右子树上所有关键字的值均大于根关键字的值；
  • 左右子树又是一棵二叉排序树。

平衡二叉树

  • 平衡二叉树首先是一棵二叉排序树，但是其要满足左右子树高度之差的绝对值不超过 1。
  • 平衡二叉树的左右子树也是平衡二叉树。

B-Tree

B-Tree 能加快数据的访问速度。

  • 因为存储引擎不再需要进行全表扫描来获取数据，
  • 数据分布在各个节点之中。

B+Tree

【概述】

  • B+Tree 是 B-Tree 的改进版本，数据都在叶子节点上，并且增加了顺序访问指针，每个叶子节点都有指向相邻的叶子节点的指针。
  • 相比 B-Tree 来说，进行范围查找时只需要查找两个节点，进行遍历即可。而 B-Tree 需要获取所有节点，相比之下 B+Tree 效率更高。

【B+Tree性质】

  • 非叶子节点不用来保存数据而是保存数据的索引。
  • 所有的叶子结点中包含了 ① 全部关键字的信息，② 及指向含这些关键字记录的指针，并且叶子结点本身按关键字的大小从小到大顺序链接。
  • 所有的非叶子结点都起到了索引的作用，结点中仅含其子树中的最大（或最小）关键字。
  • B+Tree 中，数据对象的插入和删除仅在叶节点上进行。
  • B+Tree 有 2 个头指针，一个是树的根节点，一个是最小叶子节点的头结点。

【初始化介绍】

一颗 B+树，浅蓝色的块我们称之为一个磁盘块，可以看到每个磁盘块包含了数据项（深蓝色所示）和指针（黄色所示）。如磁盘块1 包含数据项 17 和 35，包含指针 P1、P2、P3。

  • P1 指向数据项小于 17 的磁盘块，P2 指向数据项在17~35 之间的磁盘块，P3 指向数据项大于 35 的磁盘块。
  • 真实的数据存储在于叶子节点中，即 3、5、9、10、13、15、28、29、36、60、75、79、90、99。
  • 非叶子节点只不存储真实的数据，只存储指引搜索方向的数据项。如 17、35 并不真实存在于数据表中。

【查找过程】

  1. 如果要查找数据项 29，那么首先会把磁盘块 1 中的数据加载到内存中，发生第一次 IO，在内存中用二分查找确定 29 在17~35 之间，锁定磁盘块 1 的 P2 指针；
  2. 通过磁盘块 1 的 P2 指针的磁盘地址把磁盘块 3 中的数据加载到内存中，发生第二次 IO，在内存中用二分查找确定 29 在 26~30 之间，锁定磁盘块 3 的 P2 指针；
  3. 加载磁盘块 8 中的数据到内存中，发生第三次 IO，同时内存中做二分查找找到 29，结束查询，总计三次 IO。

【小总结】

在实际生产中，3 层的 BTree 可以表示上百万的数据，如果上百万的数据查找只需要三次 IO，性能提高将是巨大的，如果没有索引，每个数据项都要发生一次 IO，那么总共需要百万次的 IO，显然成本非常非常高。

Hash 索引

【原理】

基于哈希表实现，对于每一行数据，存储引擎都会根据其索引列计算一个哈希值（HashCode），并且将所有的哈希码存储在索引表中，同时在索引表中保存指向每个数据行的指针。

【缺点】

对于 Hash 索引而言，只有精确匹配索引所有列的查询才有效。

  • 这句话的意思是：对于复合索引，包含两个或两个以上的列。比如现在有一个复合索引包含了（id，name，age）三列，而 Hash 索引实现时是把这三列看做一个整体计算出一个 HashCode，所以想要利用 Hash 索引加快查找速度时，必须同时根据这三列算出的 HashCode 才是一个有效的 HashCode，如果仅仅包含了其中的一列或者两列，得到的 HashCode 就是一个非法的 HashCode，最终得到的结果自然也是不正确的。

如果 MySQL 判断全表扫描⽐使⽤索引查询更快，则不会使⽤索引。

  1. 函数导致索引失效：
  • 函数导致索引失效的原因是：索引表中保存的数据表中的字段值，如果对索引字段使用函数，在进行检索时，就需要把索引表中的每个索引都进行一遍函数运算，然后再用函数运算得到的结果去与 where 条件比较，显然成本太大，这会导致索引失效。、
  • 正确的做法是，将函数放在筛选条件上而不是索引列上。
  1. 运算符导致的索引失效：

原因与函数导致索引失效一样，解决方法是将运算过程放在等号后面的条件上而不是索引列上。

  1. OR引起的索引失效：

OR 导致索引并不是一定会发生。如果OR连接的是同一个字段，那么索引不会失效，反之索引失效。

  1. 模糊搜索导致索引失效：
  • 模糊搜索导致索引失效也并不是一定会发生的。
  • 当 % 或者 _ 放在匹配字段最前面是不走索引的，放在后面才会走索引。索引设计的时候就是从最左边的字符串开始匹配的，所以如果左边的字符串是确定的，就会走索引。

Q&A

1、为什么索引结构默认使用B+Tree，而不是B-Tree，Hash，二叉树，红黑树？

B-Tree 的缺点：

  1. B+Tree 的磁盘读写代价更低。B+Tree 的非叶子节点并没有保存指向关键字具体信息的指针，因此其非叶子节点相对 B-Tree 更小。也就是同一磁盘块能够容纳的节点数量越多，即同一磁盘块盘块所能容纳的关键字数量越多，一次性读入内存的需要查找的关键字也就越多，那么相对 IO 读写次数就降低了。
  2. 由于 B+Tree 的数据都存储在叶子结点中，分支结点均为索引，方便扫库，只需要扫一遍叶子结点即可。但是 B-Tree 因为其分支结点同样存储着数据，我们要找到具体的数据，需要进行一次中序遍历按序来扫，所以 B+Tree 更加适合区间查询的情况，所以通常B+树用于数据库索引。

Hash 的缺点：

  • 虽然可以快速定位，但是没有顺序，不支持排序；
  • 适合等值查询，如 =、in()、<=>，不支持范围查询；因为 Hash 索引中的数据也是无序的，如果想要得到有序结果的话，同样需要全盘扫描。
  • 因为不是按照索引值顺序存储的，就不能像 B+Tree 索引一样利用索引完成排序 ；
  • Hash索引在查询等值时非常快 ；
  • 因为 Hash 索引始终索引的所有列的全部内容，所以不支持部分索引列的匹配查找 ；
  • 如果有大量重复键值得情况下，哈希索引的效率会很低，因为存在哈希碰撞问题 。

二叉树缺点：

  • 查找效率不稳定。二叉树的查找效率跟树的高度有关，并且二叉树不能自平衡，如果出现某些极端数据，可能会导致二叉树的查询效率下降到接近于一个单链表。

红黑树缺点：

  • 树的高度随着数据量增加而增加。

2、非聚簇索引一定会回表查询吗？

  • 不一定，这涉及到查询语句所要求的字段是否全部命中了索引，如果全部命中了索引，那么就不必再进行回表查询。一个索引包含（覆盖）所有需要查询字段的值，被称之为”覆盖索引“。
  • 举个简单的例子，假设我们在员工表的年龄上建立了索引，那么当进行 select score from student where score > 90 的查询时，在索引的叶子节点上，已经包含了score 信息，就不会再次进行回表查询。

3、创建索引时需要注意什么？

  • 索引字段应该设置非空约束。索引应该指定列为 NOT NULL，除非你想存储 NULL。在 MySQL 中，含有空值的列很难进行查询优化，因为它们使得索引、索引的统计信息以及比较运算更加复杂。在使用时应该用 0 或者 空字符串 或者 其它特殊值代替 NULL。
  • 索引字段应该取离散程度高的字段：如果是单列索引，应该尽量选取离散程度高的字段作为索引。如果是复合索引，应该尽量让离散程度高的索引列放在复合索引的前面。可以通过 count() 函数查看字段的差异值，返回值越大说明字段的唯一值越多，字段的离散程度越高。

4、哪些情况下需要创建索引？

  1. 主键天生就是唯一索引；
  2. 频繁作为查询条件的字段应该创建索引；
  3. 查询中与其它表关联的字段，外键字段应该建立索引；
  4. 查询中排序的字段。排序字段若通过索引去访问将大大提高排序速度；
  5. 单键/组合索引的选择问题，组合索引性价比更高；
  6. 查询中统计或者分组字段；

5、哪些情况不需要创建索引？

  1. 表记录太少；【表记录过少的话遍历查找也能很快就找到所有数据，而为此再专门维护一张索引表是不合适的】
  2. 频繁更新的字段（经常增删改的表或者字段）不适合创建索引；【因为每次更新不单单要更新数据库表记录还会更新索引表记录】
  3. Where 条件里用不到的字段不创建索引（不用做查询条件的字段）
  4. 过滤性不好的不适合建索引。
  • 假如一个表中有 10w 条记录，有一个字段 gender 只有 0 和 1 两种值，且每个值的分布概率大约为 50%，那么对这种字段建立索引一般不会提高数据库的查询速度。
    • 索引的选择是指索引列中不同值的数量与表中记录数的比。如果一个表中有 2000 条记录，表索引列有 1980 个不同的值，那么这个索引的选择性就接近于 1，这个索引的效率越高。

6、建索引的原则有哪些？

  1. 最左前缀匹配原则

MySQL 会一直向右匹配直到遇到范围查询（>、<、between、like）就停止匹配，比如 a = 1 and b = 2 and c > 3 and d = 4 如果建立(a,b,c,d) 顺序的索引，d 是用不到索引的，如果建立 (a,b,d,c) 的索引则都可以用到，a,b,d 的顺序可以任意调整。

  1. = 和 in 可以乱序

比如 a = 1 and b = 2 and c = 3 建立 (a,b,c) 索引可以任意顺序，MySQL 的查询优化器会帮你优化成索引可以识别的形式。

  1. 尽量选择离散度高的列作为索引

计算离散度的公式是 count(distinct col)/count(*)，表示字段不重复的比例，比例越大我们扫描的记录数越少，唯一键的离散度是1，而一些状态、性别字段可能在海量数据面前区分度接近于 0。一般情况下索引字段的离散度应该在 0.1 以上，即平均 1 条索引扫描10条记录。

  1. 索引列不能参与计算

索引列不能参与计算，保持列“干净”，比如 from_unixtime(create_time) = ’2014-05-29’ 就没有使用到索引，原因很简单，b+Tree 中存的都是数据表中的字段值，但进行检索时，需要把所有元素都用函数计算后才能比较，显然成本太大。所以语句应该写成 create_time = unix_timestamp(’2014-05-29’)。

  1. 尽量扩展已有的索引，不要新建索引。

比如表中已经有 a 的索引，现在要加 (a,b) 的索引，那么只需要修改原来的索引即可。

7、使用索引查询一定能提高查询的性能吗？

  • 通常通过索引查询数据比全表扫描要快。但是我们也必须注意到它的代价。
  • 索引需要空间来存储，也需要定期维护， 每当有记录在表中增减或索引列被修改时，索引本身也会被修改。 这意味着每条记录的INSERT，DELETE，UPDATE将为此多付出 4，5 次的磁盘I/O。 因为索引需要额外的存储空间和处理，那些不必要的索引反而会使查询反应时间变慢。

8、联合索引及最左匹配原则

MySQL 可以使用多个字段同时建立一个索引，叫复合索引。但是在使用复合索引的时候，需要注意最左匹配原则。比如：

假设现在建立了 idx_test(id, name, age) 的复合索引，那么索引的排序为：先按照 id 排序，如果 id 相同，则按照 name 排序，如果 name 也相同，则按照 age 进行排序。

当进行查询时，此时索引仅仅按照 id 严格有序，因此必须首先使用 id 字段进行等值查询，之后对于匹配到的列而言，此时可以使用 name 字段用作索引查找，以此类推。因此在建立联合索引的时候应该注意索引列的顺序，一般情况下，将查询需求频繁的列放在前面。

总结：

  • 如果条件里含有联合索引的第一个字段，那么不管 where 条件的顺序和个数，这次查询都会走索引。
  • 联合索引的使用和 and 条件的顺序无关。
  • 只要列中包含有 NULL 值都将不会被包含在索引中，复合索引只要有一列含有 NULL 值，那么这一列对于复合索引就是无效的。

联合索引的好处：

  1. 减少了建立索引的开销。建立一个 (a,b,c) 的复合索引，实际上等于建立了 (a)，(a,b)，(a,b,c) 三个索引，因为每多一个索引，都会增加写操作的开销和磁盘空间的开销，对于大量数据的表，这是一笔不小的开销。
  2. 覆盖索引。对于复合索引 (a,b,c)，如果有如下的 select a,b,c from table where a=1，那么 MySQL 可以直接通过遍历索引取得数据，而无需会标。这将减少很多 IO 操作。
  3. 缩小筛选范围。对于 1000W 条数据的表，如果有如下 select * from table where a=1 and b=2 and c=3，假设每个条件可以筛选出 10% 的数据，如果只有单值索引，那么通过该索引只能筛选出 1000W*10%=100W 条数据，然后再回表从 100W 条数据中找到符合 b=2 and c=3 的数据，然后再排序、再分页。如果是复合索引，通过索引就可以筛选出 1000W * 10% * 10% * 10% = 1W，然后再回表查询这 1W 条数据即可。

10、主键设计 UUID 和 自增主键的区别

  • 自增主键，在进行数据库插入时，位置相对固定（B+树的右下角），数据插入效率要高一些，能更好的支持区间查找
  • UUID：由于 UUID 是随机生成的，插入时位置具有不确定性，插入效率要低一些。区间查找也没有自增主键性能优越。

链表

优点：

  1. 在插入或删除操作上效率要高于数组。
  2. 在进行检索时，效率仍然较低，因为没有下标，所以每次查找只能通过遍历的方式。

树

  1. 能提高数据存储、读取的效率。比如利用二叉排序树(Binary Sort Tree)，既可以保证数据的检索速度，同时也可以保证数据的插入，删除，修改的速度。

线索二叉树基本介绍

空指针域计算：

如果一棵二叉树有 n 个节点，那么其必然有（n-1）条连线（即必须使用的指针），按照每个节点有 2 个指针来算，未使用的空指针域共有：2n - (n-1) = (n+1) 个。

  1. n 个结点的完全二叉树中含有 n+1 个空指针域，利用二叉树中的空指针域，存放该结点在某种遍历次序下的前驱和后继结点的指针，这种附加的指针称为”线索“。
  2. 这种加上了线索的二叉数称为线索二叉树(Threaded BinaryTree)，根据线索性质的不同，线索二叉树可分为前序线索二叉树、中序线索二叉树和后序线索二叉树三种。
  3. 一个结点的前一个结点，称为前驱结点。
  4. 一个结点的后一个结点，称为后继结点。

堆排序

堆是什么？

堆是一种特殊完全二叉树的。具有以下特性：

  1. 每个结点的值都大于或等于其左右孩子结点的值，称为大顶堆。
  2. 每个结点的值都小于或等于其左右孩子结点的值，称为小顶堆
  3. 注意：没有要求结点的左孩子与右孩子值的大小关系。

堆排序

堆排序是利用堆这种数据结构而设计的一种排序算法。

  • 堆排序是一种选择排序；
  • 它的最坏，最好，平均时间复杂度均为 **O(nlog^n^)**；
  • 它也是不稳定排序。

基本思想

堆排序的基本思想是：

  1. 将待排序的无序序列构造成一个大顶堆，此时整个序列的最大值就是堆顶节点；
  2. 然后将堆顶元素与末尾元素进行交换，此时末尾元素就是最大值。
  3. 之后忽略掉末尾元素，将剩下的 n-1 个元素重新构造成一个大顶堆，得到其中的最大值（也就是全部 n 个元素中的次小值）。
  4. 如此反复执行，便能得到一个有序序列了。
  5. 如果一个堆有 n 个元素，那么只需要执行 n-1 次堆的重构即可，因为 n-1 次后就只剩下一个元素了，最后一个元素的位置是确定的。
  6. 可以看到在堆排序的过程中，每轮需要比较的元素的个数是在逐渐减少的，因为每次形成一个大顶堆后，就会将一个元素放到其最终位置上。

赫夫曼树

是什么？

基本介绍

  1. 给定 n 个权值作为 n 个叶子结点，构造一棵二叉树，若该树的带权路径长度（WPL）达到最小，称这样的二叉树为最优二叉树，也称为**赫夫曼树(Huffman Tree)**。
  2. 赫夫曼树是带权路径长度最短的树，权值较大的结点离根较近。

一些概念

  • 路径
  • 路径长度
  • 结点的权
  • 带权路径长度

如何构建Huffman树？

给你一个数列 {13, 7, 8, 3, 29, 6, 1}，要求转成一颗赫夫曼树。

构成赫夫曼树的步骤：

  1. 将数列按从小到大的顺序进行排序。每个数据都是一个节点，每个节点可以看成是一颗最简单的二叉树。
  2. 取出数列中权值最小的两个节点与一个根节点组成一颗新的二叉树，该新的二叉树的根节点的权值是前面两颗二叉树根节点权值的和。
  3. 再将这棵新二叉树的根节点权值加入数列中再次从小到大排序，不断重复 1-2-3 的步骤，直到数列中，所有的数据都被处理，就得到一颗赫夫曼树。

原理

  1. 定长编码：按照字符的内容，逐一将其转化成二进制码，传递给计算机去执行的编码方式

是什么

AOP；Aspect Oriented Programing；面向切面编程

  • 面向切面编程：通过预编译方式或运行期动态代理实现程序功能的统一维护的一种技术。
  • AOP 可以对业务逻辑的各个部分进行拆分，提高程序的可重用性，降低业务逻辑各部分之间的耦合度。
  • AOP就是把我们程序中重复的代码抽取出来，在需要执行的时候，使用动态代理技术在不修改源码的基础上，对我们已有的方法进行增强。
  • 简而言之，就是通用代码复用！

我更愿意称 面向切面编程 为面向业务编程，何为面向业务编程？每个大的业务都可以划分为粒度更小的业务，在很多个大的业务中，肯能有一些小的业务需要被重复使用，比如：日志业务、登录验证业务等。由于这些小的业务模块需要在很多大业务模块中被使用，为了避免在每个大业务模块中都编写这类重复的代码，我们可以把这些可能会多次用到的小业务模块抽取出来单独管理，在大业务模块需要使用时动态的“织入”进去，这就是所谓的 AOP（面向切面编程）。

*听完上面的话，你是不是觉得这跟抽取一个公共方法没有什么两样？那么 AOP 和抽取公共方法相比最大的优势是什么呢？就是动态代理！！！抽取公共方法的方式虽然也提高了利用率，但是需要修改原先业务逻辑中的源代码，即如果某个模块需要使用到日志功能，那么它必须将日志模块抽取成的公共方法写到业务逻辑中，而动态代理则没有这个缺点，它不需要修改原先业务的代码逻辑，而是通过一个代理对象来调用原先的业务逻辑，并将增强的方法写在代理对象中，不会修改原业务逻辑*

AOP的作用及优势

作用

  • 不修改原代码就能对对已有方法进行增强。（即不会影响原来的业务逻辑）

优势

  • 减少重复代码
  • 提高开发效率
  • 维护方便

AOP的实现方式

  • 动态代理技术

Spring中的AOP

  • Spring中的AOP，就是通过配置的方式，实现动态代理的功能。
  • AOP 就是纵向的编程，比如业务 1 和业务 2 都需要一个共同的操作（使用同一段代码），甚至可能未来的业务 n 也会使用到这段代码，为了提高代码的复用性，就会把这段代码抽取出来，让多个业务共同使用这段代码。AOP 就是把所有公共的代码抽取出来，放置到某个地方集中管理，然后在具体运行时，再由容器动态织入这些共有代码。

Spring中AOP的细节

6.1.1、AOP中的相关术语

1、Joinpoint（连接点）

  • 所谓连接点就是指那些可以被拦截到的点。
  • 在Spring中，这些点指的是方法，因为Spring只支持方法类型的连接点。
  • 【所谓的拦截点(方法)，就是指可以被增强的方法，只有拦截到这些方法后才能对其增强】【可以理解为被代理对象中的方法】

2、Pointcut（切入点）

  • 所谓切入点是指我们要对哪些Joinpoint进行拦截。
  • Pointcut一定是Joinpoint，但是joinpoint不一定是Pointcut。
  • 【在代理对象中，我们可以对代理的方法进行增强，也可以选择不增强，按照代理前的方法输入。但是这些方法都是被拦截到的点。 我们把被拦截并增强的方法叫做Pointcut，而只是完成了拦截的点叫做Joinpoint】

3、Advice（通知/增强）

  • 所谓通知就是指拦截到Joinpoint之后所要做的事情就是通知。
  • 通知的类型：
    1. 前置通知
    2. 后置通知
    3. 异常通知
    4. 最终通知
    5. 环绕通知
  • 【通知就是我们编写的增强方法，在不修改源码的基础上新增功能的代码】

4、Introduction（引介）

  • 引介是一种特殊的通知，在不修改原代码的前提下，Introduction可以在运行期为类动态的添加一些方法或Field。

5、Target（目标对象）

  • 被代理的对象

6、Weaving（织入）

  • 是指把增强方法应用到目标对象（被代理对象）来创建代理对象的过程。
  • Spring采用动态代理织入，而AspectJ采用编译期织入和类转载期织入。

7、Proxy（代理）

  • 一个类被AOP织入增强后，就产生一个代理类。

8、Aspect（切面）

  • 是切入点和通知（引介）的结合。
  • 【只有切入点没有通知或只有通知没有切入点都无法实现增强，只有切入点+通知，才能完成一次织入。完成织入的方法叫做一个切面】

6.1.3、学习Spring中的AOP要明确的事

  1. 开发阶段
    • 编写核心业务代码；
    • 把公用代码抽取出来，制作成通知；
    • 在配置文件中，声明切入点与通知间的关系，构建成一个切面。
  2. 运行阶段（Spring框架完成的）
    • Spring框架监控切入点方法的执行。
    • 一旦监控到切入点方法被调用，就使用代理机制，动态创建目标对象的代理对象，根据通知类别，在代理对象的对应位置，将通知对应的功能织入，完成完成的代码逻辑运行。

Spring TX；Spring事务管理

Spring TX 概要

  • Spring 事务管理也是 AOP 的一种实现。
  • 事务管理是一个通用的功能，可以将这个功能抽取出来，形成一个通知类（增强类）。
  • 此时，我们就可以在程序中的任何地方将这个通知织入进去，形成切面。此时，切入点方法就已经被我们的事务管理器管理了。

基于 XML 配置文件的事务管理

1 创建事务管理器

  • Spring 框架为我们提供了管理事务的对象 DataSourceTransactionManager，这个对象中集成了事务的开启、提交、回滚、关闭等操作。
  • 使用前需要先将 DataSourceTransactionManager 对象添加到 IoC 容器中。
  • DataSourceTransactionManager 对象依赖于 DataSource 对象。

2 配置事务管理器的通知（增强）

  • 事务管理器的通知和普通 AOP 中的通知原理是一样的，只不过为了更加语义化，提供了 <tx:advice> 标签来专门定义事务管理器的通知。
  • 事务管理器的通知（简称“事务的通知”）是对事务管理器的增强，增强的方式是动态代理，可以自行选择是否需要增强。
  • 事务的通知的主要作用就是在通知中动态地给事务管理器添加一些额外的功能。比如：可以在事务的通知中定义事务的属性。

2.1 配置事务的属性：<tx:attributes>

事务的属性被配置在事务的通知中，被 <tx:attributes> 标签包裹。

2.1.1 <tx:method>

<tx:method> 标签主要用于管理指定的方法。包含如下属性：

  • name：指定要被事务管理的方法的方法名，可以模糊指定或精确指定。【优先级：精准匹配>部分匹配>完全模糊匹配】
  • isolation：用于指定事务的隔离级别。默认值是 Default，表示使用数据库的默认隔离级别。【MySQL 和 Oracle 不同】
  • propagation：用于指定事务的传播行为。默认值是 REQUIRED。
  • read-only：用于指定事务是否只读。只有查询方法才能设置事务为 true。默认值是false，表示读写。
  • timeout：用于指定事务的超时时间。默认值是-1，表示永不超时，如果指定了数值，以秒为单位。
  • rollback-for：用于指定一个异常，当产生该异常时，事务回滚，当产生其它异常时，事务不会滚。没有默认值，表示任何异常都会滚。
  • no-rollback-for：用于指定一个异常，当产生该异常时，事务不回滚。产生其它异常时，事务回滚，没有默认值，表示任何异常都回滚。

3 配置 AOP 切面

  1. 找到切入点：即要对哪个（些）方法施加配置好的事务。
  2. 将事务的通知织入到切入点，形成切面

Tip：

  • 一般情况下，AOP 中的切面使用 <aop:aspect> 标签；
  • 而在事务管理中，AOP 切面则使用语义化更加明确的 <aop:advisor> 标签。

4 总结（重要）

  • 为了保证代码的可读性和易维护性，我们在 Service 层编写的对数据库的操作都是屏蔽了事务操作的、简单易懂的代码。
  • 而把对事务的控制交给了 Spring 的 AOP 去控制。[等价于使用代理对象]
    • 所以 Spring 配置文件中的切入点就是 Service 层中实现类，再通过事务的属性 <tx:method name=""> 定位到实现类中的具体方法，给对应的方法添加配置文件中配置好的事务；

所以只要将 事务的通知 织入到 切入点 中，就可以完成事务的控制。【把事务的通知织入切入点 就是 把事务相关的方法插入到业务方法中执行 的意思。】

@Transactional 注解

作用：该注解的属性与 XML 配置文件中的事务属性的含义一致。

作用范围：该注解可以在接口上、类上和方法上使用。

  • 在接口上使用，表示该接口的所有实现类都使用该事务；
  • 在类上使用，表示该类中的所有方法都使用该事务；
  • 在方法上使用，表示该方法使用该事务。
  • 以上 3 个位置的优先级：方法> 类> 接口。即如果类上声明了事务A，类中的方法上声明了事务B，则该方法以事务B为主。

事务的传播行为

1 是什么？

  • 事务的传播行为：当一个方法运行在被另一个事务管理的方法中时，必须指定事务该如何传播。例如：方法可能继续在现有事务中运行，也可能开启一个新事务，在新事务中运行。
  • 事务的传播行为可以由传播属性 propagation 指定。Spring 中定义了 7 种事务传播行为。
传播属性 描述
REQUIRED（默认） 如果有事务正在运行，当前方法就在这个事务内运行，否则，就启动一个新的事务，并在新的事务内运行。
REQUIRES_NEW 当前方法必须启动新事务，并在它自己的事务（新事务）内运行。如果有事务正在运行，应该将其挂起。
SUPPORTS 如果有事务正在运行，当前方法就在这个事务内运行；否则它可以不运行在事务中。
NOT_SUPPORTED 当前方法不应该运行在事务中，如果有运行中的事务，就将其挂起。
MANDATORY 当前方法必须运行在事务内部，如果没有正在运行的事务，就抛出异常。
NEVER 当前方法不应该运行在事务中，如果有正在运行的事务，就抛出异常。
NESTED 如果有事务在运行，当前方法就应该在这个事务的嵌套事务内运行。否则，就启动一个新的事务，
并在它自己的事务内运行。

2 怎么用？

2.1 XML配置文件中使用

  • 事务的传播属性可以在 xml 配置文件中配置。

2.2 注解中使用

  • 事务的传播属性可以在 @Transactional 注解的 propagation 属性中定义。

3 Demo

我们来模拟一个场景：现在数据库中有 100 本书，每本书 60 元，模拟客户买书，每次只能买一本，每次买书都需要付一次钱，可以买多次。客户共有 100 元，一直买书直到买不起为止。

  • 假设买书方法 buyBook() 和支付方法 pay() 都使用默认的事务传播行为 Required。在这种情况下，内部方法 pay() 和外层方法 buyBook() 用的是同一个事务，这种情况下会发生：多次买书的行为要么全部成功，要么全部失败（事务的原子性）。

    就比如在本例中，第一次买书后，客户余额 -60，库存 -1，pay() 方法正常结束，但是第二次买书时由于余额不足以支付，所以支付失败，这时候需要回滚，由于内层方法和外层方法使用的是同一个事务，所以会直接回滚到库存 100，用户余额 100 的状态。

  • 如果把事务传播行为改成 REQUIRES_NEW，在这种情况下，用户第一次买书后，就已经完成了一次事务，已经将数据库中的数据修改并提交了，即使第二次买书失败，也只会回滚到第二次买书前的状态（事务的隔离性），即库存 99，用户余额 40 的状态。

事务的隔离级别

0 前置知识

数据库事务并发导致的问题：

  1. 脏读。事务 A 读到了事务 B 更新但未提交的数据。
  2. 不可重复读。（多次获取）事务两次读到的数据不一致。
  3. 幻读。（多次获取）事务两次读到表数据的数量不一致。

隔离级别

  1. 读未提交。允许事务 A 读取事务 B 未提交的修改。
  2. 读已提交。要求事务 A 只能读取事务 B 已提交的修改。
  3. 可重复读 。确保一个事务可以多次从一个字段中读到相同的值。即事务 A 执行期间禁止其它事务对这个字段进行更新。【字段上锁】
  4. 串行化。确保一个事务可以多次从一张表中读到相同数量的行。即在事务 A 执行期间，禁止其它事务对这张表进行添加、更新、删除操作。可避免任何并发问题，但性能差。【表上锁】

1 怎么用？

在 Spring 中，通过 isolation 属性来设置事务的隔离级别。

1.1 在 xml 配置文件中设置隔离级别

1.2 在注解中设置隔离级别

2 Spring 中事务的隔离级别

  • Spring 中事务的隔离级别与使用的数据库产品相关联，比如 Oracle 就只支持两种隔离级别，而 MySQL 支持 4 种隔离级别。
  • Spring 中默认使用的是数据库产品默认的隔离级别。

比如：

  • 如果 Spring 连接的数据库产品是 MySQL，则默认的事务隔离级别是：Isolation.REPEATABLE_READ，可重复读。MySQL 默认的隔离级别。
  • 同理可得，Isolation.READ_COMMITTED，读已提交。Oracle 默认的隔离级别。

什么是事务失效？

  • 数据库事务满足 ACID 四大特性，只要破坏了其中某一特性就可以说事务失效了。

Spring 事务失效的几种场景

  2. 目标方法自己吞了异常导致事务不能正确回滚
  3. AOP 切面顺序导致事务不能正确回滚
  4. 非public方法导致事务失效
  5. final/public 方法导致事务失效
  6. 父子容器导致事务失效
  7. 本类方法调用

1 抛出编译期异常导致事务不能正常回滚

事务失效的场景：

  • 加了 @Transactional 注解的方法，理论上就加上了事务，当代码中出现异常的时候，要求符合事务的原子性，也就是说由于程序出现异常而终止时，必须对已经完成的操作进行回滚。
  • 但是在在上述代码片段的第 7 行中，new FileInputStream("aaa") 会出现编译期异常，并且在这里我们没有自己处理异常，而是直接将异常抛给了上一层，所以该代码在运行的时候也会由于异常未被处理而终止程序。
  • 但是从结果来看：from 账户余额会减少，但是 to 账户余额并没有增加。也就是说这里出现了事务失效。

事务失效的原因：

  • Spring 默认只会对出现了 RuntimeException 和 Error 的子类异常进行回滚操作，如果出现的异常是编译期异常，那么当发生异常导致程序中断时，数据是不会回滚的。

解决思路：

  • 覆盖 Spring 的默认配置，让 Spring 对所有因异常导致的程序终止都进行回滚操作。

解决方案：

  • 配置 @Transactional 注解中的 rollbackFor 属性，显式地声明出现了哪些异常后要进行回滚操作。

2 目标方法自己吞了异常导致事务不能正确回滚

事务失效的场景：

  • 目标方法内部自己处理了异常（通过 try...catch 代码块）导致事务不能正确回滚。

事务失效的原因：

  • Spring 中的事务是通过 AOP 动态织入的，也就是说，类似于 commit、rollback 等操作不是在目标方法内执行的，而是在目标方法代理对象中的同名方法中执行的。当发生了异常后，如果我们在目标方法中自己处理掉了异常，那么在同名的代理方法中，就会认为目标方法是没有发生异常的，自然也就不会触发异常回滚操作了。
  • 在业务层通过 @Autowired 注解得到的某个实例对象（比如 bean）其实是一个代理对象，在调用 bean.transfer 方法时，调用的是代理对象的 transfer 方法，代理对象的 transfer 方法中存在各种通知通知（前置/后置/环绕/异常通知），最内层的核心方法才会调用目标对象的 transfer 方法。也就是说，如果我们在目标对象的业务逻辑中直接将异常捕获并处理了，那么在代理对象的角度看来，目标对象的 transfer 方法是没有抛出异常的，自然也就不会对事务进行回滚了。

解决思路：

  • 既然在代理对象必须先发生异常然后才能进行回滚，那么我们只需要直接将目标对象的异常向上抛出给代理对象就可以了。

解决方案：

  • 可以自行选择是否在目标方法中处理异常，但是如果自己处理了异常，还需要向上层代理对象抛出这个异常信息，目的就是让代理对象也能感知到出现了异常，以便于进行回滚。

3 AOP 切面顺序导致事务不能正确回滚

事务失效的原因：

对于配置了多层 AOP 切面的方法，比如 transfer 方法有两层切面，最外层是一个事务通知的切面，内层是一个自定义的日志切面，最内层才会执行目标对象的 transfer 方法。假设 transfer 方法抛出异常，首先会被日志切面接收，如果日志切面自己处理了异常且没有将异常继续向上层抛出，那么在事务通知切面看来，程序是正常执行的，自然也就不会对事务进行回滚了。

解决思路：

  • 让事务切面可以感知到发生了异常。

解决方案：

  1. 同情况 2，可以在处理了异常的情况下，继续向上层切面抛出异常。
  2. 可以通过 @Order 注解调整切面的优先级，让事务切面成为紧挨着目标对象的切面，这样目标对象一旦抛出异常就能第一时间被事务切面捕捉到，当事务切面捕捉到发生了异常后就会执行回滚操作。

4 非public方法导致事务失效

事务失效的场景：

事务失效的原因：

  • Spring 默认要求被代理的方法必须是 public 修饰的。
  • 而 @Transactional 底层的实现原理就是为目标方法创建一个代理对象以及代理方法，如果目标方法的权限修饰符是非 public 的，创建代理方法就会失败，也就等于 @Transactional 注解失效了，自然事务也就会失效了。

解决方案：

  1. 将方法改为 public
  2. 覆盖 Spring 的默认配置，让 非public 方法也能被代理。

5 final/public 方法导致事务失效

事务失效的原因：

  • Spring 事务底层是通过 AOP 实现的，也就是通过 JDK 动态代理或 cglib，帮助我们生成了代理类，在代理类中实现事务的功能。
  • 但是如果某个方法被 final 修饰了，那么在它的代理类中，就无法重写该方法，也就无法再代理方法中添加事务的功能。
  • 同样，如果某个方法是 static 的，同样无法通过动态代理，变成事务方法。

6 父子容器导致事务失效

事务失效的原因：

  • 子容器扫描范围过大，把未加事务配置的 service 扫描进来。
  • 如果一个 Spring 程序中含有多个容器（以两个为例），且这两个容器间建立了父子关系。
  • 在发生依赖注入时，对于子容器，子容器首先从自身寻找要注入的对象，如果在自己的容器中找不到，再继续去父容器中寻找。

举个例子：

  • 在 Spring 中，如果想要让事务生效必须在配置类上使用 @EnableTransactionManagement 开始声明式事务，并且会在配置 类上规定包扫描的范围，这个范围也就是声明式事务起作用的范围。
  • 一般我们对于事务的处理都放在 Service 层进行，假设现在将 Service 层中的所有对象加入到父容器中。
  • 再创建一个子容器，子容器的扫描范围包括 Controller 层、Service 层和 Mapper 层（大于父容器），并且没有在子容器中使用声明式事务。
  • 当我们在 Controller 注入 Service 层的对象时，由于子容器中能够找到 Service 层的对象，因此它就不会去父容器中寻找了，但是子容器中的所有 Service 层的对象，都是没有声明事务的，那么用这些对象去调用带有 @Transactional 注解的方法，也不会产生事务 。

解决方案：

  1. 控制字容器的扫描范围，与父容器各自分工扫描，不要图简便将子容器范围扩大。
  2. 不要用父子容器，所有的 bean 都放在同一容器。

6 本类方法调用导致事务失效

事务失效的场景：

  • 在方法 A 中调用了方法 B，方法 A 的传播行为是 REQUIRED，即 A 中的 B 方法如果没有事务，则复用 A 的事务，如果 B 有自己的事务，则将 A 的事务挂起。
  • 现在 B 方法的使用的事务传播行为是 REQUIRES_NEW，即 B 方法会创建自己的事务。所以，理论上当调用 A 方法时，必须创建两个事务。
  • 但是在执行过程中，只有 A 方法的事务创建成功了，B 方法没有执行事务的创建。也就是说 B 方法上的事务失效了。

事务失效的原因：

  • 本类方法调用不经过代理，因此无法增强。
  • 事务失效的原因在于动态代理未生效。@Transactional 注解的实现原理是通过动态代理的动态织入让普通的方法具备事务的增强功能，我们通过 @Autowired 注解得到的可以调用 A 方法的对象就是一个代理对象，但是由于方法 A 和方法 B 位于同一个类中，在 A 方法中直接通过 this.B() 调用 B 方法是一种直接调用，this 对象是一个原始对象而不是代理对象，并没有经过对应的增强，自然也就不会创建新的事务了。

解决方案：

  • 解决思路：不要让 this 对象来调用 B 方法，而是让代理对象来调用 B 方法。
    • 方案1：用 @Autowired 注入自身的代理对象，用代理对象代替 this 对象调用 B 方法。
    • 方案2：通过 AopContext 获取当前类的代理对象，用代理对象代替 this 对象调用 B 方法。

事务失效的场景：

在转账操作中，如果有多个线程同时运行 transfer 方法，由于 transfer 方法是没有任何同步措施的，所以可能会出现多个线程同时转走了from 账户内的余额，导致 from 账户余额为负数的情况。

假设 from 账户现有 1000 元，线程 A 和 B 同时执行把这 1000 元转账到 to 账户的操作。

  1. A 线程先查询账户余额，得到 from 账户余额为 1000 元，满足转账条件；
  2. 此时 A 线程挂起，B 线程也查询账户余额，得到 from 账户余额为 1000 元，也满足转账条件；
  3. 这时候 A 和 B 线程就都会执行转账操作，最终导致 from 账户余额为负数，这在业务上显然是不允许的。

一般情况下我们会选择加锁的方式来解决这个问题。但是加锁的位置必须要注意，如果加锁的位置不当，就有可能会导致事务失效。

是什么

代理模式；Proxy；

代理模式是：为目标对象提供一个代理对象，通过代理对象操作对目标对象，目标对象只实现核心功能，其它功能都通过代理对象来实现。即通过代理对象访问目标对象。通俗的来讲代理模式就是我们生活中常见的中介。

好处：可以在不修改目标对象的情况下，扩展目标对象的功能。

被代理的对象可以是：远程对象、创建开销大的对象或需要安全控制的对象。

代理模式有不同的形式，主要有两种：

  1. 静态代理
  2. 动态代理：
    • JDK 代理（接口代理）
    • Cglib 代理：可以在内存动态的创建对象，而不需要实现接口

为什么要使用代理模式

  • 中介隔离作用：在某些情况下，如果客户类不方便直接引用一个委托对象，而代理类对象可以在客户类和委托对象之间起到中介的作用，其特征是代理类和委托类实现相同的接口。

  • 开闭原则，增加功能：代理类除了是客户类和委托类的中介之外，我们还可以通过给代理类增加额外的功能来扩展委托类的功能，这样做我们只需要修改代理类而不需要再修改委托类，符合代码设计的开闭原则。代理类主要负责为委托类预处理消息、过滤消息、把消息转发给委托类，以及事后对返回结果的处理等。代理类本身并不真正实现服务，而是同过调用委托类的相关方法，来提供特定的服务。真正的业务功能还是由委托类来实现，但是可以在业务功能执行的前后加入一些公共的服务。例如我们想给项目加入缓存、日志这些功能，我们就可以使用代理类来完成，而没必要打开已经封装好的委托类。

1 为什么使用线程池？（优势）

线程池的主要工作是控制并行线程的数量，运行过程中将待执行的任务加入队列，由线程池分配线程来执行这些任务，如果任务数量超过了最大线程数量，超出数量的任务排队等候，等线程执行完其它任务，再从等待队列中取出执行。

使用线程池的优势：

  1. 降低资源消耗。 通过重复利用已创建的线程降低线程创建和销毁造成的消耗。
  2. 控制最大并发数：线程是稀缺资源，如果无限制的创建，不仅会消耗系统资源，还会降低系统的稳定性，并发数量过多可能会导致服务器崩溃。（主要原因）
  3. 提高线程的可管理性：使用线程池可以对已创建的线程进行统一的分配、调优和监控。
  4. 提高响应速度。当任务到达时，任务可以不要等待线程创建就能立即执行。

2 怎么用

当我们使用线程池后，执行任务时就不再需要为某个任务分配某个具体的线程了，我们只需要把任务交给线程池即可，线程池内部会分配一个空闲的线程去执行该任务。

2.1 架构说明

线程池是通过 Executor 框架实现的，该框架用到了 Executors、Executor、ExecutorService、ThreadPoolExecutor 这几个类。

  • 线程池的顶级父接口是 Executor，次一级父接口是 ExecutorService，这两个接口中定义了线程池实例必须要实现的公共的方法。
  • ThreadPoolExecutor 是线程池的具体实现类，可以通过这个类来创建一个线程池对象。
  • AbstractExecutorService 是一个抽象类，是 ThreadPoolExecutor 的父类。

2.3 源码分析

查看底层源码发现，这三个创建线程池的方法底层都使用了 ThreadPoolExecutor 类。

  1. newFixedThreadPool
  • 创建一个定长的线程池。
  • newFixedThreadPool 创建的线程池 corePoolSize 和 maximumPoolSize 值是相等的，
  • 底层使用的数据结构是 LinkedBlockingQueue。
  • 执行长期的任务，性能较好

  1. newSingleThreadExecutor
  • 创建一个只包含一个线程的线程池，只会用唯一的线程来执行任务，保证所有任务按照指定顺序执行。
  • newSingleThreadExecutor 将 corePoolSize 和 maximumPoolSize 都设置为 1，
  • 底层使用的数据结构是 LinkedBlockingQueue。
  • 适用于一个任务一个任务执行的场景

  1. newCachedThreadPool
  • 创建一个自适应长度的线程池，即多退少补。
  • newCachedThreadPool 将 corePoolSize 设置为 0，将 maximumPoolSize 设置为 Integer.MAX_VALUE，也就是说来了任务就创建线程执行任务，当线程空闲超过 60s，就销毁线程。
  • 底层使用的数据结构是 SynchronousQueue。
  • 适用于执行很多短期异步的小程序或者负载较轻的服务器。

3 ThreadPoolExecutor 构造方法中的 7 大参数

ThreadPoolExecutor 有多个重载的构造方法，方法中的参数为 5~7 个不等，我们直接来看最多的 7 个参数。

  1. corePoolSize：线程池中常驻核心线程数
    • 线程池中的常驻线程数，即使没有任务时也需要去维护的线程数量。
      • 当有请求任务进来后，首先会安排常驻线程去执行任务；
      • 当线程池中正在使用的线程数量达到 corePoolSize 后，再有新的任务进来，就会把新进来的任务放到缓存队列中。
  2. maximumPoolSize：线程池能够容纳的最大线程数，此值必须大于等于 1。
  3. keepAliveTime：非核心线程的心跳时间。如果非核心线程在 keepAliveTime 内没有运行任务，非核心线程就会被会销毁。
    • 如果线程池中的线程数量超过 corePoolSize，并且多余线程空闲的时间超过 keepAliveTime 值后，多余的空闲线程将会被销毁直到只剩下 corePoolSize 个线程。
    • 只有当前线程池中的线程数大于 corePoolSize 时 keepAliveTime 才会生效，直到线程池中的线程数不大于 corePoolSize。
  4. unit：keepAliveTime 的时间单位。
  5. workQueue：阻塞队列，保存被提交但尚未执行的任务。
  6. threadFactory：生成线程池中线程实例的工厂，一般使用默认的即可。不同的工厂都能生产线程实例，就类似于联想和戴尔工厂都能生产电脑，但具体的生产过程是不一样的。
  7. handler：拒绝策略。当阻塞队列满了并且工作线程大于等于线程池的最大线程数时如何拒绝新的请求任务加入的策略。
    • 线程池的所有线程都已经在工作了，并且等待执行的任务队列也满了，这时候就需要拒绝新的任务加入，采用的策略就叫做拒绝策略。

在创建了线程池后，线程池会等待提交过来的任务请求。每次调用 execute() 方法执行新的任务请求时，线程池中会进行如下判断：

  1. 如果线程池中正在运行的线程数量小于 corePoolSize，那么马上分配一个空闲线程执行这个任务；
  2. 如果线程池中正在运行的线程数量大于等于 corePoolSize，首先会尝试将这个任务放入阻塞队列；
  3. 如果这时阻塞队列也满了但正在运行的线程数量小于 maximumPoolSize，就会创建非核心线程来执行这个任务；
  4. 如果队列满了且正在运行的线程数量大于等于 maximumPoolSize，那么线程池会执行饱和拒绝策略。
  5. 当一个线程执行完一个任务时，它会从阻塞队列中选取下一个任务进行执行。
  6. 当一个线程无事可做超过一定的时间（keepAliveTime）时，线程池会判断：
    • 如果当前线程池中运行的线程数量大于 corePoolSize ，那么这个线程就会被销毁；
    • 所以线程池的所有任务完成后它最终会收缩到 corePoolSize 的大小。

小总结：

任务首先放到线程池的核心池中，放不下就放在阻塞队列，再放不下就对线程池扩容，直到扩容到最大。这时候如果继续有很多线程要进入，启动拒绝策略；如果这时候线程开始逐渐减少，就把线程池中超过空闲时间的线程销毁掉，直到线程池中线程数等于核心线程数。

5 线程池的4种拒绝策略

当线程池中的任务数已经达到了当前线程池负载的极限时，再有新的任务交付给线程池时，线程池就会采用拒绝策略。以下是线程池中常见的四种拒绝策略：

  1. AbortPolicy（默认）：直接抛出 RuntimeExecutionException 异常，阻止系统正常运行。

  2. DiscardOldestPolicy：抛弃阻塞队列中等待最久的任务，然后把当前任务加入等待队列中，尝试再次提交当前任务。

  3. DiscardPolicy：直接丢弃当前任务，不做任何处理也不抛出异常。如果允许任务丢失，这是最好的一种方案。

  4. CallerRunsPolicy：交给“调用者“执行当前任务，该策略既不会抛弃任务，也不会抛出异常，而是将这个任务退回给调用者，让调用者线程去执行这个任务。

    注解：假设 main 线程是线程的调用者，main 线程将一个任务 A 分配给线程池处理，但此时线程池中的运行着的线程已经达到了 maximumPoolSize，即无法再给任务 A 分配一个新的线程了，需要拒绝这个任务。如果采用的拒绝策略是 CallerRunsPolicy，那么任务 A 不会被丢弃，也不会出现异常，而是会将任务 A 会退给 main 线程，让 main 线程去执行任务 A 。

以上 4 种内置的拒绝策略的实现类均实现了 RejectedExecutionHandler 接口。

6 线程在实际开发中的使用（阿里巴巴 Java 开发手册）

  1. 在实际开发中，线程资源必须通过线程池提供，不允许开发人员在程序中自行显式地创建线程。

    注解：使用线程池的好处是减少在创建和销毁线程上所消耗的时间以及系统资源的开销，解决资源不足的问题。如果不使用线程池，有可能造成系统创建大量同类线程而导致消耗完内存或者“过度切换上下文线程”的问题。

  2. 实际开发中，线程池不允许通过 Executors 工具类去创建，而是要通过创建 ThreadPoolExecutor 实例的方式创建自定义的线程池，这样的处理方式让开发人员更加明确线程池的规则，规避资源耗尽的风险。

    注解：Executors 创建的线程池对象的弊端如下：

    1. FixedThreadPool 和 SingleThreadPool：

      这两种线程池维护的阻塞队列虽然是有界队列，但是允许的最大长度为 Integer.MAX_VALUE（由 LinkedBlockingQueue 的底层数据结构决定的），这个值实际上非常大，可能会堆积大量的请求，从而导致 OOM。

    2. CachedThreadPool 和 ScheduledThreadPool

      这两种线程池虽然使用的请求队列是同步阻塞队列，最大容量为1，但是允许创建的最大线程数量为 Integer.MAX_VALUE，也可能会创建大量的线程，从而导致 OOM。

7.3 如何合理配置线程池参数

CPU密集型任务

CPU 密集型也是指计算密集型，大部分时间用来做计算逻辑判断等 CPU 操作的程序称为 CPU 密集型任务。该类型的任务需要进行大量的计算，主要消耗 CPU 资源。这种计算密集型任务虽然也可以用多任务完成，但是任务越多，花费在任务切换上的时间就越多，CPU 执行的效率也就越低，所以想要最高效的，所以，想要最高效地利用 CPU，CPU 密集型任务同时进行的数量应当等于 CPU 的核心数。

  • CPU 密集的意思是该任务需要大量的运算，而没有阻塞，CPU 一直全速运行。
  • CPU 密集任务只有在真正的多核 CPU 上才能得到加速（通过多线程）
  • 而在单核CPU上，无论你开几个模拟的多线程该任务都不可能得到加速，因为CPU总的运算能力有限。
  • CPU 密集型任务的配置尽可能少的线程数量。
  • 一般公式：CPU 核数 + 1个线程的线程池

IO密集型

IO 密集型指任务需要执行大量的 IO 操作，涉及到网络、磁盘 IO 操作，相对 CPU 消耗较少。

  • IO 密集型，即该任务需要大量的 IO，即大量的阻塞。
  • 在单线程上运行 IO 密集型的任务会导致大量的 CPU 运算能力浪费在等待。
  • 所以在 IO 密集型任务中使用多线程可以大大的加速程序运行。即在单核 CPU 上，这种加速主要就是利用了被浪费掉的阻塞时间。
  • 由于 IO 密集型任务并不是一直在执行任务，则应该配置尽可能多的线程，如 CPU核数×2

知识点补充

execute() vs. submit()

execute() 方法

  • 用于提交不需要返回值的任务，所以只能提交实现了 Runnable 接口的任务，并且无法判断任务是否执行成功；

submit() 方法

  • submit() 方法既可以提交实现了 Runnable 接口的子任务，也可以提交实现了 Callable 接口的子任务。
  • 用于提交需要返回值的任务。线程池会返回一个 Future 类型的对象，通过这个 Future 对象可以判断任务是否执行成功，并且可以通过 Future#get() 方法来获取返回值，get() 方法会阻塞当前线程直到任务完成，而使用 get(long timeout，TimeUnit unit) 方法则会阻塞当前线程一段时间后立即返回，这时候有可能任务没有执行完。

线程池常用的阻塞队列有哪些?

  • 表格左侧是线程池，右侧为它们对应的阻塞队列，可以看到 5 种线程池对应了 3 种阻塞队列

  1. LinkedBlockingQueue
  • 对于 FixedThreadPool 和 SingleThreadExector 而言，它们使用的阻塞队列是容量为 Integer.MAX_VALUE 的 LinkedBlockingQueue，可以认为是无界队列。
  • 由于 FixedThreadPool 线程池的线程数是固定的，所以没有办法增加特别多的线程来处理任务，这时就需要 LinkedBlockingQueue 这样一个阻塞队列来存放任务。
  • 这里需要注意，由于线程池的阻塞队列永远不会放满，所以线程池只会创建核心线程数量的线程，所以此时的最大线程数对线程池来说没有意义，因为并不会触发生成多于核心线程数的线程。
  1. SynchronousQueue
  • 第二种阻塞队列是 SynchronousQueue，对应的线程池是 CachedThreadPool。线程池 CachedThreadPool 的最大线程数是 Integer.MAX_VALUE，可以理解为线程数是可以无限扩展的。
  • CachedThreadPool 和上一种线程池 FixedThreadPool 的情况恰恰相反，FixedThreadPool 的情况是阻塞队列的容量是无限的，而这里 CachedThreadPool 是最大线程数可以无限扩展，所以 CachedThreadPool 线程池并不需要一个任务队列来存储任务，因为一旦有任务被提交就直接转发给线程或者创建新线程来执行，而不需要另外保存它们。
  1. DelayedWorkQueue
  • 第三种阻塞队列是 DelayedWorkQueue，它对应的线程池分别是 ScheduledThreadPool 和 SingleThreadScheduledExecutor，这两种线程池的最大特点就是可以延迟执行任务，比如说一定时间后执行任务或是每隔一定的时间执行一次任务。
  • DelayedWorkQueue 的特点是内部元素并不是按照放入的时间排序，而是会按照延迟的时间长短对任务进行排序，内部采用的是“堆”的数据结构。之所以线程池 ScheduledThreadPool 和 SingleThreadScheduledExecutor 选择 DelayedWorkQueue，是因为它们本身正是基于时间执行任务的，而延迟队列正好可以把任务按时间进行排序，方便任务的执行。

Executor 和 Executors 的区别？

  • Executors 工具类是一个用于创建常用线程池对象的工具类，提供了创建常见线程池的方法。
  • 而 Executor 接口是线程池对象的顶级接口，内部定义了一个 execute() 方法用于执行我们的线程任务。ExecutorService接口继承了Executor接口并进行了扩展，提供了更多的方法，比如：我们能获得任务执行的状态并且可以获取任务的返回值。
  • 使用 ThreadPoolExecutor 可以创建自定义线程池。Future 表示异步计算的结果，他提供了检查计算是否完成的方法，以等待计算的完成，并可以使用 get() 方法获取计算的结果。
