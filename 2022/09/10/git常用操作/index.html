Git基础

暂存区的作用

暂存区的作用：暂时存放，还没有作为一个正式的版本提交到仓库中。

但是需要注意的是：加入暂存区的文件就开始被Git管理了，暂存区中的文件可以很容易地被Git变成一次正式的提交；同时，它又可以回退，用暂存区的文件覆盖工作目录的文件，在实际开发中有很多使用场景。

比如：

  1. 在工作目录中执行修改操作引发的变更，可以先添加到暂存区；
  2. 然后再在工作目录中尝试使用另外一套解决方案，假如第二套方案没有第一套方案好的话，就可以用暂存区的第一套方案覆盖工作目录。

config

config 是一个配置文件，保存了Git的一些配置信息。对于这些配置信息，我们既可以通过Git提供的一些简单命令进行直接修改（比如用户名和邮箱信息），也可以在配置文件上进行修改，最终效果都是一样的。

refs

refs 目录下有 heads 和 tags 两个目录，复数意味着Git仓库是可以有很多 head 和 tag。

  • tag 又叫做“里程碑”，记录了某一次具有重大意义的commit。比如我们的项目开发到了一个新的阶段，需要对当前的commit进行发版，比如 V1.0，这时候可以对当前最新的 commit 打上一个标识（tag）。但实际上，tag 引用的也是这次 commit 的 id。
  • heads 对应着我们的许多分支，分支就是独立的开发空间，对于一个人来说，比如我们开发前后端分离的项目，我们可以为前端建立一个分支，为后端建立一个分支，两个分支相互独立开发。对于多人协作开发来说，每个人可以在自己独立的分支上进行代码开发，开发完毕后将所有的分支合并到主分支上，避免开发过程中频繁冲突的问题。

实际上，不管是 heads 还是 tags 中，保存的都是一些文件。对于 heads/master 文件，文件内容指向的是 master 分支下最新一次的commit_id，对于 tags/xx 文件，文件内容指向的是发布 tag 那一次的commit_id。

我们来具体看一下这个 id 的类型，可以看到它确实是一个 commit 类型。

objects

Git中的Object可以参考Java的Object，是最顶级的父类。包括commit、tree、blob、branch、tag 等都可以看成是一个 Object。

objects 中都是一些目录。可以看到有非常多的文件夹，一个比较特殊的文件夹是 pack，pack 的作用就是整理，当 objects 中文件夹的数量太多时，Git 就会进行自我梳理，将一些前缀一样的文件打包整理到 pack 中

以文件夹 e6 为例，这里面存放的都是以 “e6” 开头的 object，我们通过将 “e6” 作为前缀与文件夹内的子文件夹名称进行拼接，可以得到一个完成的 id，根据这个 id 可以查看对应的 object 类型。比如下面我们就查出了两种类型 commit 和 blob，实际上 Git 中还有类型为 tree 的 object。

commit、tree和blob三个对象之间的关系

Git对象彼此关系

commit

每次执行 git commit 命令都会创建一个 commit 对象。一个 commit 一定会对应于一棵树（tree）。

这棵树表示这次 commit 后得到的当前项目所有文件夹及文件的快照视图。

  • parent：上次提交的commit_id
  • tree：保存本次提交的所有数据的快照的tree对象的id
  • author：作者
  • committer：提交者

tree

在 tree 对象中，我们还可以看到子 tree 对象，表示文件夹，对应的，blob 对象表示文件。

  • tree：目录
  • blob：文件

blob

文件内容。

注意的一点是：blob的唯一id与文件名没有一点儿关系。只要文件的内容相同，那么 Git 就会将其视为一个 blob。所以对于一些名字不同而内容相同的文件，Git 不会对内容进行重复存储，只需要将这两个文件名指向同一个 blob 即可。

分离头指针情况下的注意事项

什么是分离头指针？

分离头指针：HEAD指针脱离分支而独立存在。即commit没有基于某个分支（branch），这就会导致HEAD不与分支绑定，这种情况就叫做“分离头指针”。

分离头指针的使用场景

通常情况下，我们都会使用 git checkout <branch_name> 命令来进行分支间的切换，但是 checkout 后面不光可以跟分支名称，还可以是某次的 commit_id 号，比如下面这样，我们使用“第二次提交”的 commit id 作为 checkout 的目的地。

与正常切换分支不同的是，这次切换我们可以看到一大段的建议信息，大意如下：

  • 你现在正处于“分离头指针”状态下，你可以在这个状态下进行一些实验性的修改，如果你觉得实验结果好的话，可以提交这些修改，如果觉得实验结果不好的话，可以放弃在这个状态下进行的任何提交。通过切换回现存的某个分支来放弃这些修改。
  • 如果你想保留你在这个状态下的提交，需要创建一个新的分支，你可以参考如下命令：

给它一个使用场景就是：我们现在想基于某一个 commit 进行实验性的开发，那么就可以进入“分离头指针”状态，在一个没有分支的状态下工作，在这个状态下进行试验开发并提交代码。如果试验的结果不理想，你不想要这些测试代码了，那么你可以直接切回到一个现有的分支下（比如 master），你在 “detached HEAD” 状态下进行的所有开发都会在不远的将来的某一时间被 Git 清理掉。

当然，如果你觉得这次试验很成功，那么一定不要忘了给 “detached HEAD” 绑定一个真正的分支，否则这些提交都会被 Git 当做无关紧要的 commit 而丢弃掉。

分离头指针的使用案例

通过前面一系列工作，我们现在已经处于 “detached HEAD” 状态，当前数据是基于“第二次提交”这个commit。我们现在修改 css 文件，将其中的某个参数的颜色改为 green，然后通过 git status 查看工作区与暂存区的状态。可以看到Git已经识别出了，当前处于分离头指针状态，并且对应于某一个 commit。

继续 commit 这次修改，并查看 log 状态。可以看到，在 log 中，HEAD 指针并没有指向某个具体的分支，这与我们前面看到的是不一样的，正常的应该类似于 HEAD -> master，这也是“分离头指针”名字的由来，即HEAD指针与分支分离。

我们查看所有分支上的 log 树可知，Git 并没有把 “detached HEAD” 状态下的提交当成一次正式提交，所以当切换到某个具体分支后，在 “detached HEAD” 状态下的提交都会在将来某个时间段被 Git 清理掉。

比如，我们在进行分支切换的时候，Git 也会提示你：如果你想保存这次提交的话，可以为 “detached HEAD” 状态下的提交创建一个新的分支。

我们可以先为其创建一个新的分支，然后重新查看所有分支的 log 记录，此时，“detached HEAD” 状态下的分支由于绑定了具体的分支，这些提交也就变成了有效提交，可以被永久保存。

总结

如果你想进行一些尝试性的变更，可以在分离头指针的状态下进行，如果尝试的结果不理想，可以直接 checkout 到一个开发分支上，那么你在分离头指针状态下的变更会被自动回收。

当进行分支切换时，在分离头指针状态下产生的切换很可能会被Git当做垃圾回收掉，如果你认为这些变更是重要的，切记一定要将commit与分支进行绑定。

如果你想进行变更，请一定要跟某个分支关联。在分支的基础上，对分支进行变更。

HEAD的特殊标识符

几种常见的表示方式：

  • HEAD~1：HEAD 的上一个版本
  • HEAD~2：HEAD 的前两个版本
  • HEAD^1：等价于 HEAD~1
  • HEAD^：等价于 HEAD^1
  • HEAD^^：HEAD 的前两个版本，等价于 HEAD~2

怎么删除不需要的分支

如图所示，我们正处于 master 分支下，准备删除 fix_readme 分支，执行 git branch -d fix_readme 命令后出错，错误信息是：

怎么修改老旧commit的message？

git rebase——变基

  • 变基：这里的“基”是指某一次commit，“变基”是指改变这次commit之后的所有commit。
  • 如果执行 git rebase 成功，那么“基”之后的所有commit_id，都会使用新生成的commit_id。

注意事项：

rebase——变基，这里的“基”指的是“基于谁”。所以如果想要改变某一次的 commit_message，首先要找到这次commit的父commit，基于 parent_commit_id 进行rebase操作。

需要注意的是：执行变基操作后，这个“基”后面的所有commit都会变更，也就是说，如果我们以“第三次提交”的commit 进行 rebase 操作，那么其后的所有操作都会发生改变，而不是说只改变紧跟其后的一次commit。

默认对每次 commit 的操作就是 pick，即“选用”（其真正的含义是将某个commit的信息选出来加入到分离头指针状态下的commit中），但是我们现在的需求是：改写 2f79a92 的信息，那我们就需要执行 r, reword 这条命令。

在这个文件中，我们只需要选择策略上的修改，即 pick -> reword，在这里不可以进行 message 的修改。在我么修改完保存后，我们会进入到另外一个界面，在另一个界面上真正完成对 commit_message 的修改。

修改 commit_message

看到如下提示信息，就说明修改成功了。我们可以看到，这里用到了“分离头指针”的操作，我们搭配 log 看一下这个操作。

首先，Git会根据 rebase（变基）命令后变基的文件中的所有command命令，在本例中就是一个 pick 加一个 reword，并将这两个 commit 作为分离头指针状态下的commit。然后再将分离头指针状态下的 commit 合并到 master 分支上。自然，合并到master分支时使用的 commit_id 是在分离头指针状态下的 commit_id，所以原来的“基”(33a29b6) 之后的所有 commit_id 都会改变，自然也就需要更改 HEAD 指针的位置。

需要注意的是，虽然 commit_id 发生了变化，但是这次 rebase 并没有改变文件的内容，所以对于新的 commit 来说只是改变了相应的 tree 对象和 blob 对象的引用地址，变动其实并不是很大。

怎样把间隔的几个commit整理成一个？

当然，为了防备后期还有可能需要方案A的代码，我们可以对工作区的数据先进行一次 stash 操作，记录它当前的快照。

开发中临时加塞了紧急任务怎么处理：stash

场景

  • 你的上一个功能正在开发中，一部分代码被提到了暂存区，一部分代码还在工作区正在编辑；
  • 但是，突然来了一个优先级比较高的需求，需要在其它分支上处理紧急出现的bug；
  • 此时，你就需要先将你完成的部分工作保存到某一个地方，去处理其它分支上的bug，等处理完毕后切回自己的开发分支再继续开发。

可以看到，当前工作区的styles/style.css 以及暂存区的 images/git-logo.png 和 index.html 与 HEAD 相比都发生了变化，此时有一个新的任务加塞进来，我们就可以使用 git stash 命令将这些变更暂时保存到一个栈区域，生成一个栈元素，这个栈元素保存了这些变更是基于哪一次的 commit。

可以使用 git stash list 来查看 stash 区域保存的变更，我们可以看到，stash 区域保存的变更是基于所有分支的。

将这些变更加入到 stash 区域后，工作区和暂存区就恢复到了 HEAD 的状态，之后，你就可以切换分支去完成需要处理的紧急任务了。

完成紧急任务之后，我们想回溯之前未完成的任务，可以使用 git stash apply 或 git stash pop。

  • git stash apply 的作用是将 stash 区域的内容取出来并添加到工作区，但不会删除stash区域的记录。
  • git stash pop 的作用与 apply 类似，但 pop 是一个弹出栈的动作，除了将 stash 区域的内容取出来并添加到工作区后，还会会删除stash区域的记录。
